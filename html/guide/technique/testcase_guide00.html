목적 및 구성

1. 목적 및 범위

본 문서는 블랙박스 테스트(Black-Box Test) 수행에 필요한 Test Case를 도출하고 작성하는 방법을 설명한다.

최종 고객의 기대사항에 대하여 검증하고 확인하여 사용자에게 명확하게 제시하는은 사용자 만족도를 크게 제고할 수 있다.

Test Case는 확인되어야 하는 요구사항이 반영된다.

그러나 모든 요구사항에 대하여 확인하지 못할 수도 있으므로, 비용과 위험 그리고 필요성에 맞게 가장 적당하고 핵심적인 요구사항을 테스트 대상으로 선정하는 것이 중요하다.

Test Case는 다음과 같은 역할을 한다.

  • 테스트 절차 개발 및 설계 기초
  • 테스트 깊이는 Test Case 수에 비례. 각 Test Case는 다른 시나리오와 조건 그리고 어플리케이션 내 흐름을 반영하고 있기 때문에 Test Case 수가 많을 때 어플리케이션과 테스트 절차에 대한 보다 큰 확신을 얻을 수 있다.
  • 테스트 완전성에 대한 기본적인 측정방식은 요구사항 대비 수행된 테스트 범위임. “핵심 테스트 95%가 수행되고 확인되었다” 것이 “우리가 테스트의 95% 수준에 있다.”보다 중요하고 의미가 있다.
  • 테스트를 위한 노력의 크기는 Test Case의 수에 비례.
  • 다양한 테스트 디자인과 설계 그리고 필요한 자원들은 Test Case에 의해 좌우

2. 구성

Test Case 작성 가이드 내용은 다음 항목으로 구성된다.

  • 작성시기 및 담당자 : 문서 작성 시기 및 작성 담당자 명시
  • 주요 입력물 : 문서 작성 시 필요한 참조 문서목록
  • 작성 절차 : Test Case 작성 순서
  • 도구 사용시 고려사항 : 테스트 자동화 도구 사용 시 고려해야 할 사항에 대한 설명
  • Test Case 도출기법 : 테스트 유형별 Test Case 도출 기법에 대한 설명
  • 산출물 구성 : Test Case모델의 목차 및 내용 설명

1.1. 작성시기 및 담당자

  • 작성시기 : 각 단계별 테스트의 설계 단계
  • 담 당 자 : 단위테스트-개발자, 통합/시스템-테스트 담당자

1.3. 일반적인 테스트 케이스 작성 절차

  • 참조 문서 수집 : 테스트 계획서에 명시된 테스트 케이스 작성지침과 수준을 고려하여 필요한 분석 및 설계문서를 수집
  • 테스트 케이스 작성 : 지침에 따라 테스트 케이스 작성
  • 내부검토 : 개발자와 테스트 담당자가 작성된 Test Case의 적정성 검토
  • 요구사항 대비 커버리지 분석 : 테스트 케이스가 어느 정도 요구사항을 반영하는가에 대한 분석. 기본적으로 테스트 가능한 요구사항은 모두 Test Case에 반영되어야 함
  • 승인 : 작성된 테스트 케이스(스위트)는 PM와 고객(현업)에게 승인을 획득한다. 일반적으로 단위 테스트 이후 고객의 승인을 득할 수 있다.
    (단위 테스트 케이스 제외)

1.4. 도구 사용시 고려사항

Test Case 도출을 자동으로 수행해주는 도구는 존재하지 않으나, 도출된 테스트 수행 편의를 위하여 테스트 자동화 도구를 이용할 수 있다.

테스트 자동화 도구를 이용하는 경우에도 테스트 스크립트(Test Script) 작성 시간을 고려하여야 하기 때문에 내부적으로 적용 기준을 설정하고 이에 적합한 Test Case만 선택하여 자동화 도구를 이용한 테스트를 실행할 수 있다.

자동화 도구 적용에 적합한 일반적인 Test Case유형은 다음과 같다.

  • 수시 업그레이드나 잦은 프로그램 변경이 발생하여 반복적으로 테스트해야 하는 경우
  • 동일한 항목에 대하여 다량의 테스트 데이터를 적용해야 하는 경우
  • 중요 업무에 대하여 객관적인 테스트 결과를 제시하야 하는 경우
  • 테스트 시 비교 대상 항목이 많은 경우

도출된 Test Case 대비 자동화 비율은 30% 이상은 넘지 않는 것이 일반적이다.

Test Case 자동화를 위해서는 Test Script를 작성해야 하며 Script를 Recording하는데 상당한 시간이 소요되며 일부분에 있어서는 프로그램까지 해야 하므로 자동화 Test Case 선정 시에 시간/비용 대비 효과를 염두에 두어야 한다.
