개요

1. 목적

본 문서는 기능테스트 수행에 필요한 지침을 제공하는데 목적이 있으며, 통합관점과 시스템관점을 포괄하고 있으나 통합관점의 테스트에 더 효율적으로 적용 가능하다.

본 문서에 기술된 지침은 테스트 수행자가 테스트 엔지니어 또는 개발엔지니어인지와는 상관없이 적용 가능하며 신규 개발 시스템과 유지보수 단계에 있는 시스템 모두에 적용 가능하다.

2. 적용 범위

본 문서는 통합테스트 수행 프로세스의 주요 TASK 설명에 중점을 두고 있다.

또한 통합 테스트 수행에 사용가능한 테스트 기법과 접근법을 포함한다.

3. 기능(통합/시스템) 테스트 정의

단위 테스트에서 개별적으로 테스트 된 모듈들은 전체 프로그램을 구성하기 위해 통합될 필요가 있다.

기능 테스트는 단위 테스트를 거친 프로그램 및 모듈을 시스템 환경과 함께 점진적으로 그룹하여 전체 시스템으로 통합될 때까지 테스트 하는 활동을 포함한다.

기능 테스트 개념은 다음의 두가지 관점에서의 정의를 모두 포함한다.

  • 통합되는 소프트웨어 단위 항목들간의 인터페이스를 테스트하여 프로그램들이 상호 정확한 커뮤니케이션을 수행할 수 있는지를 검증하고 하나로 통합된 프로그램 그룹 (인터페이스 되는 프로그램들)이 시스템 명세 내용과 일치하는지를 검증하는 것이다. (통합 관점에서의 기능 테스트 : 통합테스트)
  • 구현된 시스템이 시스템의 목표(Objectives)를 만족시키는지 판단하기 위한 테스트로, 시스템 전반적에 걸친 테스트는 시스템이 목적한 바와 같이 수행되는지 여부에 초점을 맞추고, 개발자는 시스템 테스트를 통하여 시스템의 운영 가능성에 대한 확신을 얻어야 한다. 또한 시스템의 기능적 측면과 비기능적 측면이 모두 고려되어야 한다. (시스템 관점에서의 기능 테스트 : 시스템테스트)

4. 기능 테스트 유형

통합 테스트는 단위 테스트를 마친 가장 작은 단위들간의 통합, 완성된 시스템들간의 인터페이스 테스트부터 시스템 관점에서의 기능테스트까지를 포함하는 것으로, 통합 대상의 규모/범위에 따라 여러 단계에 거쳐 반복적으로 일어나는 테스트 활동이다. 아래 리스트는 통합 테스트 유형의 몇 가지 예를 보여주고 있다.

통합관점에서의 기능테스트 : 통합 테스트

단위 테스트와 마찬가지로 특정 기법의 적용 없이 무작위로 통합 테스트 케이스를 도출하는 방식은 테스트 케이스의 품질 및 정확한 커버리지 산정이 어려우므로, 부분적으로라도 체계적인 테스트 기법을 적용, 테스트 케이스를 생성하는 것이 바람직하다. 통합 관점에서의 통합 테스트는 다음과 같은 관점에서 테스트 케이스를 도출하여 수행한다.

  • 단위 테스트가 끝난 단위 모듈들간의 인터페이스 테스트
  • 통합 중간 단계의 기능테스트
  • 전체 어플리케이션을 대상으로 기능간 연관성 테스트 (즉, 어플리케이션의 모든 프로그램들간의 인터페이스 검증)
  • 신규 어플리케이션과 기존 어플리케이션(Legacy system)과의 인터페이스 검증
  • 소프트웨어 요소들과 그 밖의 모든 시스템 환경 요소들(예, 하드웨어 요소들)과의 통합에 따른 시스템 검증
  • 웹 사이트 연결 테스트 (웹 시스템의 경우): 주로 페이지간 연결 흐름이 원활한가 테스트.
  • 지역적으로 기능이 분산되어 있는 시스템의 전체 기능 테스트 (클라이언트/서버 시스템)
  • (종종 물리적으로 떨어진) 타 시스템과의 연동 테스트

통합관점에서의 기능테스트는 진행에 따라 어플리케이션 프로그램은 통합된 기능을 가진 시스템으로 검증을 받게 되며 통합 테스트 환경은 기존 개발환경을 이용하거나, 또는 테스트 수행을 위해 특별히 필요한 장비가 있으면 별도의 테스트 환경을 구축할 수 있다.

시스템 관점에서의 기능 테스트 : 시스템 테스트

시스템 관점에서의 기능 테스트는 다음과 같은 관점에서 테스트 케이스를 도출하여 수행한다.

  • 시스템 환경에서 모든 기능적인 요구사항(Functional Requirement)이 완전하게 작동되는지 여부
  • 하드웨어, 소프트웨어, 사용자, 그리고 운영자 인터페이스 측면에서의 사용성 및 성능

이러한 전체 시스템 관점에서의 테스트는 시스템의 현실적인 검증을 위하여 실 운영시 예상되는 환경을 최대한 반영하여야 한다.

5. 통합 테스트 전략

기능 테스트 수행에 앞서 먼저 해야 할 일 중 하나는 개발시스템 구조를 정의한 산출물들을 참조하여 어떻게 시스템을 통합하고 테스트 순서를 정할 것인지를 결정하는 것이다.

이러한 전략은 비점진적 통합(Big-Bang Integration)과 점진적 통합(Incremental Integration)의 두 가지로 구분된다.

비점진적 통합은 각각의 모듈을 단위 테스트에서 독립적으로 테스트 하고, 모든 모듈에 대한 단위 테스트가 완료되면 전체 프로그램으로 한꺼번에 통합시키는 방법이고, 점진적 통합의 경우는 각각의 모듈을 독립적으로 테스트 하기 보다는 현재 단위 테스트가 끝난 모듈과 다음에 테스트 될 모듈이 통합되어 이 둘이 같이 테스트 되는 방법이다.

즉 점진적 통합 방법이 사용될 때는 통합 테스트가 별개의 테스트 단계로 생각되기 보다는 단위 테스트의 한 부분으로서 같이 진행이 된다.

점진적 통합기법의 경우는 통합 테스트가 빨리 이루어지므로 인터페이스상의 에러가 더 빨리 발견되고 발견된 문제에 대한 디버깅도 더 용이한 장점이 있지만, 비점진적 통합보다 더 많은 노력이 필요하다.

점진적 통합 전략에서는 통합의 순서를 결정하는 방법으로 상향식(Bottom-up) 방법이나 하향식(Top-down) 방법이 많이 사용된다.

하향식 통합은 소프트웨어 계층구조에서 가장 상위에 위치한 주프로그램(Main program)에서부터 시작하여 제어계층을 하향식으로 전개하면서 점진적으로 프로그램을 테스트하는 방식이고, 상향식 (Bottom-up) 통합방식은 소프트웨어 계층구조의 최하위부터 상위프로그램으로 점진적으로 프로그램을 통합하고 테스트해 나가는 방식이다.

상향식과 하향식 통합 방식은 시스템 구조 또는 프로그램 구조가 명확하게 계층형 구조(Hierarchical structure)를 보이는 기존의 시스템 (Old system)에서는 많이 적용되었지만, 최근에는 상향식과 하향식 통합 전략을 적용하는 것이 적절하지 않은 경우가 많이 있다.

예를 들어, 객체지향 시스템의 경우, 시스템 단위들이 OO 패러다임에서 지원하는 여러 관계
(예, Inheritance, Aggregation, Polymorphism 등)들로 상호 연결된 복잡한 구조를 보이는 경우가 많아, 상/하향식 통합 방법보다는 종속관계 분석(Dependency Analysis)등과 같은 새로운 기법을 통해 통합 순서를 정하는 경우가 많다.

1.1. 테스트 범위 정의

테스트 계획의 첫번째 단계는 테스트 범위를 정의하는 것이며, 테스트 범위는 테스트 종류에 따라 달라진다.

테스트 계획은 테스트 전략을 바탕으로 수립되어야 한다.

통합 테스트 대상 영역들을 테스트 목표에 맞게 각각의 통합 테스트 별로 분리하여 통합 테스트 그룹을 확정한다.

통합 테스트는 반복적으로 점진적으로 일어나는 경우가 대부분이어서 하나의 단위가 여러 통합 테스트의 대상으로 반복적으로 나타날 수 있다.

또한 테스트 범위는 기간, 예산, 비즈니스 영역 및 프로젝트 규모에 따르게 정의되어야 한다.

테스트 대상 범위 중에서도 테스트 환경이나 일정 상의 문제로 인해 테스트가 불가능한 부분이 발생할 것으로 예측되면 필수 테스트 영역, 선택 테스트 영역, 테스트 불가 영역을 선정하여 사전에 고객과 협의하고, 테스트 범위를 명확히 한 후 테스트를 실행하도록 한다

인터페이스(Interface) 요구사항 파악
  • 각 통합 그룹의 통합되는 대상 단위들에 대하여 인터페이스 관련 테스트 요구사항을 조사하여 리스트
  • 파악된 인터페이스 요구사항이 불명확한 경우 상세화 작업 수행
통합 테스트 대상 항목의 기능과 기타 추가적인 요구사항 식별
  • 통합되는 대상 항목에 대하여 기능 및 기타 비기능 (성능 특성, 상태 특성, 입출력 데이터 특성 등) 요구사항을 조사하여 리스트
  • 통합 테스트 대상 항목과 관련된 사용자 인터페이스 또는 운영 절차(operating procedures)를 식별
  • 단위 테스트에서 이미 작성된 해당 문서들을 참조하거나, 해당 시스템 산출물을 직접 분석하여 통합되는 전체 기능에 대한 새로운 요구사항 도출
    (즉, 개별 단위에서는 해당사항이 없어서 제외되었지만 통합된 전체 단위에서는 이슈가 되는 테스트 요구사항 도출)
  • 식별된 기능 및 기타 요구사항 들이 불명확한 경우 상세화 작업 수행
테스트 수행시 포함될 요소(Elements) 선정
  • 앞의 태스크에서 식별된 통합 테스트 대상 항목의 여러 요소(Elements)들 중 테스트 할 특성 선정.
    즉, 통합 테스트에 포함될 (시스템) Interface 특성, 기능 특성, 비기능 특성, 상태 특성 (States, State Transitions), 입출력 데이터 특성 (Data Characteristics), 사용자 인터페이스(User Interface) 특성 등을 선정. 입출력 데이터 선정시는 Invalid 와 Valid input 데이터를 반드시 구분하여 선정할 필요가 있음
  • 식별된 모든 요소의 테스트가 불가능할 때에는 일부를 선택해야 하는데, 이때는 예상되는 사용 빈도 등의 정보를 반영하여 선택을 결정

필요시, 테스트 범위를 기반으로 하여 다음의 테스트 수행 전략을 수립할 수 있다.

테스트 우선순위 결정: 중요도나 원활한 테스트 수행을 위하여, 또는 그 밖의 이유로 테스트 순서를 결정할 필요가 있을 때 우선 순위를 정한다.
  • 시스템의 성공적인 구축에 핵심적인 (또는 큰 영향을 미치는) 특성
  • 구현하기에 복잡하리라 예상되는 특성
  • 경험상 결함이 많이 나타나는 것으로 인식된 특성
수작업 대 자동화 테스트: 수작업으로 테스트를 해야 하는 경우도 있고 자동화가 필요한 경우도 있을 수 있으므로 수작업과 자동화 테스트에 관한 적절한 전략을 세운다.

1.2. 테스트 착수 기준 및 완료기준 정의

통합 테스트 수행 가능한 상태인 착수기준과 테스트 활동을 끝마쳐야 할 시기 결정을 위한 완료 기준 또는 종료 규칙을 정의한다.

아래 표는 통합 테스트의 착수 및 완료기준의 예이다.

착수기준 완료기준
  • 단위 프로그램의 단위 테스트 수행완료
  • 통합 테스트 케이스 검토, 승인
  • 통합 테스트 대상 SW 빌드 완료
  • 통합 테스트를 위한 테스트 환경과 테스트 데이터 준비완료
  • 계획된 모든 테스트 케이스가 성공한 경우
  • 수정되지 않은 결함이 있다 하더라도, 수정되지 않은 모든 결함이 검토되고 그러한 결함이 미미한 것으로 확인된 경우
  • 중대한 결함이 만족스럽게 조치되었고 재 테스트 결과 모두 통과한 경우
  • 발견하고 조치한 결함수가 사전에 정의한 목표에 부합되는 경우
  • 지속적인 테스트 노력에도 불구하고 새로운 오류가 거의 발견되지 않는 경우
    (관리기준선에 관한 메트릭이 존재할 경우, 예: 테스트 노력 100시간당 발견된 결함 수)

1.3. 테스트 환경 정의

테스트 환경 정의는 테스트 계획 수립의 중요한 부분이며, 테스트의 성공 여부는 테스트가 수행될 환경에 영향을 받는다. 테스트에 수행에 필요한 H/W, 시스템 S/W, 네트워크, DBMS 등의 테스트 환경을 식별한다.

테스트에 필요한 특수장비 또는 물품이 존재한다면 이를 식별한다.
예) 클립, Post-it과 같은 사소한 것에서부터 무게가 나가는 중요한 물품까지 고려할 수 있다.

필요하다면 시스템 설치방법, 데이터베이스 셋업, 필요한 스크립트나 프로그램 설치절차 및 활동, 접근권한, 백업절차를 정의한다.

프로젝트 구성관리 계획서에 본 내용이 기술되어 있다면 중복 내용에 대해서는 구성관리 계획서 참조로 대체할 수 있다.

테스트 환경은 프로젝트에서 독점적으로 사용하는 것이 바람직하나, 경우에 따라 하나 이상의 프로젝트가 공유하여 사용할 수 있다.

테스트 환경을 공유할 경우 소프트웨어 구성이 상이하여 위험을 야기할 수 있으므로, 계획 수립 시 주의해야 한다. 또한 테스트 환경과 실행 환경간의 불일치가 존재한다면, 이것이 미치는 영향이 검토될 필요가 있다.

테스트 환경 식별시 고려사항
  • 테스트 될 시스템이 운영될 환경과 구성은 어느 것인가?
  • 테스트가 수행될 플랫폼과 환경은 무엇이며, 이러한 테스트 환경이 실 운영 환경과 얼마나 일치하는가?
  • 테스트중인 시스템, 테스트 케이스와 테스트 지원 소프트웨어를 다른 플랫폼 또는 구성 하에서 어떻게 전환할 것인가?
  • 테스트에 필요한 다른 지원 시설은 없는가?
  • 어떤 테스트 데이터(DB)가 필요한가?
  • 기존 데이터가 테스트시 사용 가능한가?
  • 테스트 데이터는 어떻게 생성할 것인가?
  • 테스트 데이터를 어떻게 유지 관리할 것인가?
테스트 라이브러리/디렉토리 정의
  • 테스트를 수행할 떄 필요한 프로그램/데이터 라이브러리 및 디렉토리등의 위치를 정의한다.
테스트 데이터 생성 및 유지절차 정의
  • 테스트 데이터는 테스트의 중요한 요소로, 테스트 데이터 생성 및 관리를 위해 일반적으로 데이터 조작, 데이터 샘플링, 데이터 복제 기법이 사용된다.
기법 구분 내용
데이터 조작 약간의 데이터 변경만으로 시스템 전체를 테스트할 필요가 있을 경우 사용되며, 보고 기능이나 주요한 트랜잭션 및 시스템 전체를 테스트할 때 사용된다
데이터 샘플링 모든 테스트 데이터로 테스트 하는 것이 비용과 시간 관점에서 불가능할 경우 사용된다
데이터 복제 일반적으로 성능 테스트나 수작업으로 생성하기 힘든 크기의 테스트 데이터가 필요할 경우 사용된다.
  • 테스트하는 동안 마스터 데이터가 변경될 수 있다면, 다음 테스트 실행 전에 마스터 데이터를 저장해 두어야 한다.
    테스트 데이터 생성 및 관리를 위해 적절한 절차가 수립되어야 하며 실 데이터와 동일한 테스트 데이터가 요구될 경우, 실 테스트 데이터를 유지하기 위한 Action plan을 세우는 것이 중요하다.

(예: 필요한 테스트 자원(Resource), 통합 테스트 대상 그룹의 가용성 여부 (즉, 테스트 일정 내에 구현이 완료되어 테스트가 가능한지), 필요 자원 준비, 테스트 환경 준비, 기타 제약사항 등을 반영하여 통합 테스트 활동에 대한 상세 일정을 명세

1.1. 테스트 케이스(Test Case) 식별

명세된 테스트 대상 범위 및 요소를 “테스트 될 특성(Features to be tested)”과 선택된 관련 속성(Procedures, State Transitions, Data 특성)들에 의해 명시된 또는 암시되는 조건들을 기반으로 하여 계층형의 테스트 목표 구조를 (Test Objectives hierarchy) 테스트 케이스 명세서에 개략 설계한다.

계층 구조의 가장 하위 레벨에 있는 테스트 목표(Test Objective)는 몇 개의 테스트 케이스에 의해서 직접적으로 테스트 될 수 있도록 구성한다.

테스트 케이스는 하나로 여러 개의 프로그램이 테스트될 수 있도록 범용성을 갖게 개발하는 것이 좋다.

  • 기존 사용중인 테스트 케이스 중 적절한 것이 있다면 선택하여 재 사용
  • 테스트 케이스 식별 번호를 최하위 레벨의 테스트 목표와 연관시킨다.
  • 테스트 Lot/Case 명세서의 “테스트 케이스 ID” 컬럼에 테스트 목표 계층도와 연관된 테스트 케이스 식별자를 부여

1.2. 테스트 스텝(Test Step) 및 검증 포인트(Verification Point) 정의

테스트 케이스를 수행하는 절차를 명세. 테스트 절차는 하나의 테스트 케이스를 실행하는 일련의 과정(Steps)으로 테스트 케이스를 실행하기 위한 사전 준비 (set up), 실행 과정, 결과 분석 과정 등의 모든 단계를 상세하게 순서대로 기술한 것이다.

  • 기존의 테스트 케이스가 있을 경우, 일부 Step을 추가/삭제/변경해서 사용
  • 통합 테스트 케이스 명세서의 ‘비고’ 컬럼 등에 필요한 추가 절차 명시

경우에 따라서 하나의 테스트 케이스에 하나 이상의 테스트 절차가 있을 수 있고 반대로 여러 테스트 케이스가 하나의 절차를 공통적으로 사용하기도 한다.

경험이 많은 테스터의 경우 절차서 없이 테스트 케이스만 보고도 어떻게 테스트를 수행해야 할지 직관적으로 알 수 있는 경우도 있지만, 테스트 절차가 아주 복잡하거나 경험이 많지 않은 초보 테스터를 위해 그리고 일관성 있는 테스트 결과를 얻기 위해서는 테스트 절차를 상세하게 기술하고 문서화 하여야 한다.

1.3. 테스트 케이스 속성 정의

테스트 케이스 별로 작성자, 작성일, 테스트 케이스 ID, 설명, 기대결과 등의 속성을 테스트 케이스 템플릿에 맞게 정의한다.

  • 새로운 테스트 케이스를 명세. 필요하면 기존 명세서 참조
  • 환경적 요구사항 설정: 테스트 케이스 실행에 필요한 시스템 소프트웨어, 어플리케이션 소프트웨어와 요구되는 하드웨어 특징과 구성을 명세화하고 해당 테스트 케이스 실행보다 선행되어야 하는 테스트 케이스 식별자 나열
  • 테스트 케이스 실행 절차의 특수한 제약사항이나 환경적 요구사항을 기술
  • 통합 테스트 케이스 명세서 ‘비고’컬럼에 기록하거나 별도의 문서에 기록

1.4. 테스트 데이터(Test Data) 준비 및 검증

테스트 데이터 준비 및 검증은 상위 레벨의 테스트 케이스들을 실행 가능한 형태의 실 데이터로 바꾸는 작업(즉, 테스트 케이스에 따른 실제 입력 데이터를 준비)이다. 생성된 테스트 데이터 역시 문서화 하거나 또는 경우에 따라 테스트 케이스까지만 문서화할 수있다.

  • 이전 테스트에서 사용된 기존 테스트 데이터 이용 (변경 없이 또는 변경해서 사용)
  • 새로운 테스트 데이터 생성
  • 테스트 케이스와 테스트 데이터 Set과의 관계가 명백하지 않을 때 Correlation을 기록하기 위한 테이블을 개발하고 테스트 케이스 명세서에 별도로 포함
  • 자동화 된 테스트를 수행하는 경우, 테스트 케이스를 자동화 하는 작업 수행
  • 단위 테스트 스크립트가 있는 경우 이들을 조합. 모듈간의 연결성을 검증할 수 있는 새 스크립트 추가

1.5. 통합 테스트 케이스 개발시 고려사항

통합 테스트 케이스 개발 기법

단위 테스트와 마찬가지로 특정 기법의 적용 없이 무작위로 통합 테스트 케이스를 도출하는 방식은 테스트 케이스의 품질 및 커버리지 산정에 문제가 있을 수 있으므로 부분적으로라도 체계적인 테스트 기법을 적용, 테스트 케이스를 생성하는 것이 바람직하다.

통합 테스트 케이스 도출은 단위 테스트와 마찬가지로 Black-Box 와 White-Box 기법 모두 적용 가능하지만, 테스트 대상의 규모가 커짐에 따라 일반적으로 Black-Box 기법이 더 많이 적용된다.

통합 테스트 목적은 일차적으로 시스템 구성 항목간의 연계를 검증하는 것으로 인터페이스가 주 관심 대상이 되므로 여러 가지 흐름도 (기능 흐름, 인터페이스 흐름, 데이터 흐름)를 참조하여 실질적으로 발생될 수 있는 데이터 흐름을 식별하고 그 흐름에 따라 테스트케이스를 작성하는 것이 바람직하다. 또한 추가적으로 시스템 관점에서의 테스트 케이스를 도출한다.

필요시, 다음의 사항들을 고려하여 테스트 케이스를 개발할 수 있다.

  • 테스트 케이스 도출 기법
    • 여러 테스트 기법 중 어떤 기법을 사용하여 테스트 케이스를 도출할 것인지 정의.
      기법 선정은 현 프로젝트의 시스템 특성, 테스트 요구사항 특성, 테스트 목표, 제약사항, 테스트 기간, 현재 지원 가능한 기법인지 등을 고려하여 적용 가능한 최상의 기법을 선택 한다.
      (예를 들어 프로젝트에서 테스트 커버리지가 중요한 경우 화이트박스 기법 중 하나를 선택).
      대개 하나 이상의 기법을 혼합하여 사용하는 것이 효과적이다.
  • 생성할 테스트 케이스의 수
    • 테스트 기법, 단위의 크기, 시스템의 중요도, 테스트 기간 등을 고려해 생성할 테스트 케이스의 수를 정한다.
      (예, “적용한 테스트 기법이 제시하는 수만큼 테스트 케이스를 생성한다” 또는 “각 요구사항 항목에 대해 5개 이상 테스트 케이스를 도출한다”)
  • 테스트 케이스(데이터) 처리 방안
    • 테스트 케이스가 프라이버시/보안과 관련된 민감한 데이터인 경우 데이터 처리 전략을 명시한다.
      (예, “프라이버시를 보장하기 위해, 실 운영 파일들로부터 도출된 모든 테스트 데이터는 테스트 후 모두 삭제”)
  • 테스트 케이스 구현 계획
    • 예: 테스트 케이스 자동화 계획 수립

도출할 테스트 케이스 수 계획의 예

  • 기능 요구사항 정의서 내에 구체적 특성 당 (간단하고 위험이 낮은 특성의 경우) 2-3개의 테스트 케이스 도출. 최소의 기능 특성은 1 라인 문장으로 표현될 수 있음
  • 기능 요구사항 정의서 내에 구체적 특성 당 (복잡하고 위험이 큰 특성의 경우) 10-20 개의 테스트 케이스
  • 기능점수(Function Point) 당 2-3개의 테스트 케이스
  • 기능 요구사항 또는 명세서 문서 페이지당 20-30개의 테스트케이스
  • 조회당 2-3개
  • 조회당 10-15개 (복잡, 위험이 큰 경우)
  • 단위 화면별 5-10개 테스트케이스 (간단, 위험 낮은 경우)
  • 단위 화면별 10-25개 (복잡, 위험이 큰 경우)
테스트 케이스 개선 및 향상

생성된 통합 테스트케이스를 검토한다.

검토 후 테스트 케이스가 만족스럽지 못하거나 부족하다고 판단될 경우 추가 개발

  • 알고리즘과 내부 데이터 구조의 특성들을 고려하여 테스트 케이스를 향상
  • 고객 요구 정의서(명세서), 요구사항 설계 명세 정보를 기반으로 테스트케이스를 추가/변경하고 기록된 테스트 케이스를 업데이트 한다.
  • 고객요구사항 추적표를 기반으로 부족하다고 판단될 경우 추가 개발

통합 테스트 케이스 명세서 완성

테스트 케이스 작성시 유의사항

개발자가 자신의 프로그램에 대해 전적으로 테스트 케이스나 테스트 데이터를 준비하는 경우 중요한 결점을 찾아 내지 못할 수도 있으므로, 다른 개발자 또는 별도의 전담 테스트 조직이 테스트 케이스나 데이터를 준비할 필요가 있다.

그러나 처음에는 시스템을 설계한 설계자가 작성하는 것이 바람직하다. 테스트 케이스는 적합하고 예상되는 상황뿐 아니라 부적합하고 예상하지 못한 것들도 고려하여 작성하여야 한다.

기본적인 테스트 실행 절차

기본적인 테스트 실행 절차는 자동/수동 실행여부, 플랫폼 유형, 테스트 실행자가 누구든지간에 동일하다. 기본적인 테스트 실행 절차는 다음과 같다.

  1. 테스트 리포지토리(Repository) 또는 테스트 계획서에서 테스트 대상 케이스를 추출
  2. 해당 케이스와 관련된 의문사항 및 미해결 이슈를 해결
  3. 테스트를 성공적으로 실행하기 위하여 요구되는 초기 조건이 갖추어졌는지 점검. 준비 되지 않았다면 테스트 케이스에 맞는 테스트 환경 준비
  4. 리포지토리(Repository)/라이브러리에서 테스트 실행에 필요한 데이터 추출
  5. 테스트 데이터를 입력 후 결과 대기. 가능하다면 테스트 중간 과정과 결과를 관찰
  6. 테스트 결과 관찰 및 수집
  7. 테스트 결과 성공/실패 여부 평가
  8. 테스트 결과를 테스트 로그(테스트 케이스 명세서)에 문서화
  9. 테스트 케이스와 관련된 특징적인 증상 및 메트릭스 관련 데이터 수집
  10. 향후, 동일 테스트 케이스 재실행 시의 효과를 제고하기 위해 도움이 될 만한 사항을 기록
  11. 테스트 결과가 실패인 경우, 디버깅과 사후조치를 위하여 사건(문제) 보고서를 (결함관리대장 또는 SR시스템에) 작성
  12. 다음 테스트 케이스를 반복 실행

1.1. 테스트 환경 설정

테스트 실행을 위한 전제사항 들을 구축한다. 운영 환경에서 기능 테스트를 수행하기로 했다면 운영 환경으로 구축된 어플리케이션을 탑재한다.

  • 테스트 케이스 명세서에 기술된 테스트 수행시 필요한 테스트 지원 자원 획득
  • 경우에 따라 테스트 실행을 위해 추가적으로 요구되는 지원 프로그램(예: 시뮬레이터, DB 확인 SW 등) 구축
  • 테스트 대상 시스템 셋업 : 테스트 대상 시스템을 테스트 조건에 맞도록 초기화하여 테스트 수행 가능토록 함. 테스트 실행 전에 사전 정의된 초기 상태로 시스템을 설정해야 하고, 또한 테스트를 마친 다음에는 다음 테스트 실행을 위해 시스템 상태를 다시 초기화
  • 테스트 데이터 셋업: 테스트 데이터 (DB) 준비 및 백업
  • 필요시, 테스트 자동화 도구 준비

1.2. 테스트 케이스 실행

  • 테스트 케이스(Test Case) 실행
  • 통합 테스트를 수행하면서 발생하는 모든 테스트 사항들을 (테스트 케이스 명세서 상에) 로그(Log)로 기록
  • 통합 테스트 계획서 및 케이스 명세서에 기술된 테스트 기준 및 예상결과를 실제 테스트 결과와 비교, 일치하지 않거나 문제가 발생한 경우 해당 사항을 결함관리대장에 기록

각각의 테스트 항목에 대한 단위 테스트 기간동안의 시간대별 테스트 활동을 기록한다. 테스트 주기별, 테스트 케이스 별로 테스트 수행 내역을 기록하고, 테스트 진척상황 모니터링을 위해 활동 기록(Log)을 사용할 수 있다.

결함관리대장 작성

통합 테스트 실행 테스트 항목이 예상결과(Expected Result) 등과 비교하여 실제 결과와 차이가 발생하는 경우, 그 차이를 해당 결함관리대장에 기록한다.

결함관리대장은 테스트 활동 동안의 사건을 테스트 관리자, 프로젝트관리자 및 개발자 등에게 통보하기 위해 사용된다.

테스트 요원 및 관련자들은 테스트 수행 중 발생한 사건을 결함관리대장이 기록하고 이를 지정된 관리자에게 통보하여야 한다.

통보 받은 관리자는 해당 사건에 대한 우선순위를 부여하여 기술된 문제점을 해결하도록 조치를 취한다.

해당 프로그램의 개발자는 결함관리대장의 문제점을 해결하기 위해 결함관리대장을 분석하여 필요한 소프트웨어 변경과 문제점 원인을 파악한다.

1.2 테스트 커버리지 평가

테스트 케이스와 고객요구 정의서와의 연관관계를 요구사항 추적(요구사항 추적표 활용)하여 테스트 범위의 충분성/적절성을 판단한다.

커버리지 측정(Coverage Measures)은 “테스팅이 얼마나 완전한가” 하는 질문에 대한 답을 제공하려는 것이다. 현실적으로 100% 완전한 테스트가 불가능하기 때문에, 어느 정도 테스트가 수행 되었는지 파악하기 위해 커버리지 측정이 요구된다.

테스트 커버리지는 사전에 정의된 특정 기준(Criteria)을 기반으로 측정 하게 되는데, 일반적으로 요구사항과 코드 기반의 테스트 커버리지 (Requirements-Based & Code-Based test coverage) 측정이 가장 많이 사용된다.

테스트 커버리지 유형

  • 요구사항 커버리지 (Requirement-Based Test Coverage) : 테스트로 요구사항이 어느정도 테스트 되었는지를 측정
  • 기능 커버리지 (Function-Based Test Coverage) : 기능의 어느 정도 테스트 되었는지 측정 (하나의 요구사항과 관련된 다수의 기능이 존재할 수 있음)
  • 코드 커버리지(Code-Based Test Coverage) : 테스트로 소프트웨어의 어떤 부분이 실행되었고(커버되었고) 어떤 부분이 실행되지 않았는지 측정 (예: 구문(statement) 커버리지, 결정(decision) 커버리지, 조건(condition) 커버리지 등.

체계적인 테스트 활동을 위해 하나 이상의 테스트 커버리지 전략을 세워 테스트를 수행하는 것이 바람직하며, 테스트 평가 단계에서 테스트 실행 정도를 측정하여 해당 테스트 커버리지 기준을 만족했는지 평가할 필요가 있다.

커버리지 기준이 만족되지 못한 경우에 측정된 메트릭은 얼마나 많은 테스트 노력이 더 필요한지, 그렇다면 어느 부분이 더 테스트 될 필요가 있는지등에 대한 정보를 제공한다.

  • 테스트가 적절하게 되지 못한 부분이나 두드러지는 품질 위험 요인을 기술

  • 테스트 계확과의 불일치사항 발생시, 원인
  • 실제 테스트 수행 범위
  • 테스트 결과에 따른 평가
  • 테스트 목표, 확인기준, 위험 분석에 따른 테스트 항목의 평가

정의된 (다음 단계) 작업의 진행을 허가하는 일반적이고 특정한 조건의 집합. 시작 조건의 목적은 시작 조건을 만족하지 못한 채 테스트 되어 추후에 더 많은 (헛된)노력이 소요되는 태스크(task)를 시작하지 못하도록 하는 것이다
테스트 종료 조건(Test Exit Criteria)
절차가 공식적으로 완료되었음을 허가할 목적으로, 이해관계자들이 합의한 일반적이고 특정한 조건의 집합. 완료조건의 목적은 아직 종료하지 못한 미해결의 일부 업무가 여전히 존재 함에도 완료한 것으로 간주하려는 것을 방지하는 것이다. 완료조건은 언제 테스팅을 중지할 것인지에 대해 보고하고 계획하기 위해 사용된다. 테스트 중단을 결정하는데 필요한 전제 조건이다.
테스트 레벨(Test Level)
함께 편성되고 관리되는 테스트 활동의 그룹. 테스트 레벨은 프로젝트에서 책임과 연관 되어있다. 테스트 레벨의 예로는 단위 테스트, 통합 테스트, 시스템 테스트 그리고 인수 테스트가 있다.
테스트 접근방법(Test Approach)
특정한 프로젝트를 위해 테스트 전략을 구현하는 것. 테스트 접근법은 일반적으로 프로젝트의 목표와 리스크 평가 수행 결과를 기반으로 여러가지 테스팅 관련 의사결정을 한다. 이 밖에도 테스트 접근법은 테스트 프로세스를 시작하는 시점, 적용할 테스트 설계 기법, 완료 조건과 수행 할 테스트 유형을 결정하는 역할을 한다.
테스트 유형(Test Type)
하나 이상의 상호 연관된 품질속성에 대해 컴포넌트나 시스템을 테스팅 하고자 목표하는 테스트 활동 묶음. 테스트 유형은 신뢰성 테스트, 사용성 테스트, 회귀 테스트 등과 같은 특정 테스트 목표에 집중한다. 그리고 테스트 유형은 하나 이상의 테스트 레벨 또는 테스트 단계에서 발생할 수 있다
테스트 요구사항(Test Requirement)
하나 또는 그 이상의 테스트 케이스에 의해 검증될 수 있는 컴포넌트나 시스템의 항목 또는 이벤트. 테스트 상황의 예는 기능, 트랜잭션, 품질 속성, 또는 구조적 요소 등이다.
품질 속성(Quality Attribute)
항목의 품질에 영향을 미치는 특징(feature)이나 특성(characteristic)
개발 생명주기(Development LifeCycle)
소프트웨어 제품의 개념 형성에서 시작하여 운영/유지 보수에 이르기까지 변화하는 개발의 전 과정
동적 테스팅(Dynamic Testing)
컴포넌트나 시스템 소프트웨어를 실행하면서 수행하는 테스팅
정적 테스트(Static Testing)
리뷰 또는 정적 코드 분석과 같이 소프트웨어의 실행 없이 명세 또는 구현, 개발 단계에서 컴포넌트나 시스템을 테스팅 하는 것.
정적 테스트(Static Testing)
리뷰 또는 정적 코드 분석과 같이 소프트웨어의 실행 없이 명세 또는 구현, 개발 단계에서 컴포넌트나 시스템을 테스팅 하는 것.
테스트 스위트(Test Suite)
테스트 대상 컴포넌트나 시스템에 사용되는 여러 테스트 케이스의 집합. 테스트 스위트는 테스트 사후조건이 주로 다음 테스트를 위한 사전조건이 되는 테스트 케이스로 구성된다.
테스트 스텝(Test Step)
테스트 실행자가 최소의 교육/훈련으로 테스트를 실행할 수 있도록 테스트 실행을 위한 동작 내용 및 순서를 상세하게 기록한 것.
테스트 데이터(Test Data)
테스트가 실행되기 이전에 데이터베이스와 같은 곳에 존재하며 테스트 대상 컴포넌트나 시스템에 영향을 주거나 영향을 받는 데이터.
테스트 케이스(Test Case)
특별한 목표 또는 테스트 상황(test conditions)을 테스팅하기 위해 개발된 입력값, 실행 사전조건, 예상 결과, 실행 사후조건 들의 집합. 특별한 목표와 테스트 상황은 특정 프로그램 경로를 실행하거나 지정된 요구사항을 준수하는지 검증하는 것을 의미한다.
테스트 목적(Test Obejctive)
테스트를 설계하고 실행하기 위한 근거 또는 목적
테스트 예상 결과(Expected Result)
테스트 케이스 명세(specification) 또는 다른 소스로 예측할 수 있는 명시된 조건 하에서의 컴포넌트나 시스템의 동작.
테스트 로그(Test Log)
테스트의 실행에 대한 관련 세부항목의 시간 순서에 따른 기록
테스트 자산(Test Asset)
테스트 활동에 의해 개발된 모든 산출물을 의미한다. 예를 들어 테스트 계획서, 테스트 결과 보고서, 테스트 스크립트, 테스트 로그, 테스트 케이스, 테스트 데이터, 테스트 결과, 테스트 스위트이 있을 수 있다.
커버리지(Coverage)
특정한 커버리지 항목이 테스트 스위트(Test Suite)에 의해 이행되는 백분율 정도.
테스트 인시던트(Test Incident)
테스트 과정에서 발생하고 추가 조사가 요구되는 이벤트
동등 분할 기법(Equivalence Partitioning))
동등 분할된 영역에서 대표값으로 수행하도록 테스트 케이스를 설계하는 블랙박스 테스트 설계 기법. 원칙적으로 테스트 케이스는 각 영역을 적어도 한번은 커버할 수 있게 설계한다.
경계값 분석(Equivalence Partitioning))
동등 분할된 영역에서 대표값으로 수행하도록 테스트 케이스를 설계하는 블랙박스 테스트 설계 기법.
결정 테이블 테스팅(Decision Table Testing)
테스트 케이스가 결정 테이블(Decision Table)에 표시된 입력값과 자극(원인)의 조합을 테스트 하도록 설계하는 블랙박스 설계 기법
결정 테이블(Decision Table)
테스트 케이스를 설계할 때 사용할 수 있는 입력과 자극(원인)의 조합을 연관된 출력과 활동(결과)과 함께 보여주는 테이블.
유즈케이스 테스팅(Use Case Testing)
사용자 시나리오를 실행하도록 테스트 케이스를 설계하는 블랙박스 테스트 설계 기법
