(기능) 통합 테스트 케이스 개발 기법

1. 결정 테이블 테스팅(Decision Table testing) 기법

테스트 케이스가 결정 테이블(Decision Table)에 표시된 입력값과 자극(원인)의 조합을 테스트 하도록 설계하는 블랙박스 설계 기법.

결정 테이블이란 테스트 케이스를 설계할 때 사용할 수 있는 입력과 자극(원인)의 조합을 연관된 출력과 활동(결과)과 함께 보여주는 테이블을 말한다.

결정 테이블 테스팅 기법은 특정 유형의 시스템 요구 사항을 도출하거나 내부 시스템 디자인을 문서화하는 유용하다. 이것은 시스템이 구현해야 하는 복잡한 비지니스 규칙을 문서화하는데 사용된다.

결정 테이블 (Decision Class)

결정 테이블은 테스트 케이스를 작성하는 가이드로 활용할 수 있다. 결정 테이블은 조건 집합에 근거한 복잡한 비지니스 규칙을 나타낸다. 기본적인 형태는 다음과 같다.  

일반적인 결정 테이블 형태

  • 조건1~m까지는 다양한 입력 조건을 나타냄
  • 액션 1~n 까지는 다양한 입력 조건의 조합에 따른 액션을 나타냄. 각각의 룰은 룰과 관련된 액션의 실행 결과인 특정한 조건의 조합을 정의함
  • 액션은 평가되는 조건 순서와는 관련이 없으며, 각 조건의 값들과만 관련한다.(모든 값은 동시에 사용이 될 수 있다는 가정)
  • 액션은 특정한 조건에만 의지하게 되며, 이전 입력 조건이나 시스템의 상태에 영향을 받지는 않는다.

조건        
조건1        
조건2        
...        
조건m        
         
액션        
액션1        
액션2        
...        
액션n        
결정 테이블 (Decision Class) 테스팅 적용방법

자동차 보험 회사에서는 운전자의 결혼여부 또는/그리고 학생인지의 여부에 따라서 할인을 해준다.

이러한 조건에서 결정 테이블 테스팅 기법을 적용해본다.

아래 결정 테이블은 두 개의 조건이 있으며 각각은 Yes 또는 No 둘 중의 하나의 값만을 가진다.  

이진 조건을 가진 결정테이블

  테스트케이스1 테스트케이스2 테스트케이스3 테스트케이스4
조건        
결혼여부 Yes Yes No No
학생여부 Yes No Yes No
  • 위의 결정 테이블은 모든 가능한 조건의 조합을 포함하고 있다.  두 개의 바이너리 조건(Yes 또는 No)이 존재하며, 각 조건간의 가능한 조합은 다음과 같다. {Yes, Yes}, {Yes, No}, {No, Yes}, 그리고 {No, No}.  
  • 각 룰은 이러한 조합을 나타낸다. 테스트 요원은 정의된 모든 조건의 조합을 확인해야 한다. 일부 조합이 생략된다면 특정 입력 조건들의 조합을 제대로 처리하지 못하는 시스템 개발 가능성이 존재하게 된다. 
  • 각 조건들의 조합에 따른 액션은 다음과 같다. 각 룰은 "결과"가 발생하게 되는 원인이 되며 각각의 룰은 그 룰에 단일 액션을 나타내거나 다양한 액션이 나타날 수 있다. 

결정 테이블에 단일 액션이 추가되는 경우

  테스트케이스1 테스트케이스2 테스트케이스3 테스트케이스4
조건        
결혼여부 Yes Yes No No
학생여부 Yes No Yes No
         
액션        
할인금액 60 25 50 0
  •  결정 테이블은 각각의 룰에 대해서 한 가지 이상의 액션을 정의할 수 있다. 즉 이 룰들은 유일하거나 공유될 수 있다.

결정 테이블에 다중 액션이 추가되는 경우

  테스트케이스1 테스트케이스2 테스트케이스3 테스트케이스4
조건        
결혼여부 Yes Yes No No
학생여부 Yes No Yes No
         
액션        
액션-1 X 실행 Y 실행 X 실행 Z 실행
액션-2 A 실행 B 실행 B 실행 B 실행
  • 위의 결정 테이블에서 조건은 입력을, 액션은 예상 결과를 나타낸다.  이번 예제에서는 간단한 이진 조건을 사용하였으나, 조건은 상황에 따라 더 복잡할 수 있다.

Non-binary 조건을 가진 결정 테이블의 경우

  테스트케이스1 테스트케이스2 테스트케이스3 테스트케이스4
조건        
결혼여부 0-1 1-10 10-100 100-1000
학생여부 <5 5 6 또는 7 >7
         
액션        
액션-1 X 실행 Y 실행 X 실행 Z 실행
액션-2 A 실행 B 실행 B 실행 B 실행
  • 이런 경우 테스트 케이스 선택은 조금 복잡하다.
    각 룰(수직 컬럼)은 테스트 케이스가 될 수 있지만, 값은 이 상황은 테스트 케이스의 선택이 약간 더 복잡해 진다.
    각 룰(수직 컬럼)은 테스트 케이스가 되지만, 적절한 값을 선택하여 테스트 케이스를 만들어야 한다.

결정 테이블 기법을 적용한 Test Case 예제

테스트 케이스 ID 조건#1 조건#2 기대결과
TC 1 0 3 X 실행 / A 실행
TC 2 4 5 Y 실행 / B 실행
TC 3 50 7 X 실행 / B 실행
TC4 500 10 Z 실행 / B 실행
  • 만일 테스트 대상 시스템이 완전한 비즈니스 규칙을 가지고 있고, 비지니스 분석가나 디자이너가 이런 (또는 유사한) 형태로 규칙을 문서화하지 않았다면, 결정 테이블 테스팅 기법을 적용하기 위하여 테스터 요원은 관련 정보(조건/액션 등)를 수집하여 결정 테이블의 형태로 나타내야 한다.
    시스템 동작이 결정 테이블의 형태로 완전하고 간단히 표현될 수 있다면 결정 테이블에서 테스트 케이스를 곧바로 생성가능하기 때문이다.
  • 각 룰(수직 컬럼)에서 최소한 하나 이상의 테스트 케이스 생성한다. 
    룰이 이진 조건이라면 각 조합의 단일 테스트 케이스로 충분할 수 있다.
    만약 조건이 값의 범위라면 최소, 최대 값 두 개를 대상으로 테스팅을 고려할 수 있다.
    이런 방법으로 결정 테이블 테스팅 기법과 경계값 분석 테스팅 기법을 결합할 수 있다.  
  • 결정 테이블로부터의 테스트 케이스 명세 작성은 간단히 테이블의 열과 컬럼의 제목을 변경하고, 지정된 Template에 기술하면 된다.

Non-binary 조건을 가진 결정 테이블의 경우

  

 테스트케이스1 테스트케이스2 테스트케이스3 테스트케이스4

입력

        

조건-1

 YES YES NO NO

조건-2

 YES NO YES NO

 

        

기대결과

        

액션-1

 X 실행 Y 실행 X 실행 Z 실행

액션-2

 A 실행 B 실행 B 실행 B 실행

예제 1

  • 사용자 요구사항
    시민이 역(Station)의 자동 개찰구를 통과하기 위해서는 자동화된 승차권(1회용) 또는 정액 권이 필요하고, 승차시에 개찰이 되어 있어야 한다. 이런 전제가 충족되면 하차 후에 자동 개찰구를 통과할 수 있다. 만일 승차권이 자동화되지 않은 것이라면 역원에게 안내되고, 승차 시에 개찰이 안된 경우에는 통과할 수 없도록 개찰구가 폐쇄되어야 한다. 이와 같은 사용자 요구사항에 대하여 다음과 같이 원인과 결과를 정리가능하다.
  • 원인(조건)
    • 자동화 된 승차권
    • 자동화 된 정액권
    • 승차시 개찰되어 있음
  • 결과(액션)
    • 자동 개찰구 통과 가능
    • 역원에게 안내됨
    • 개찰구 폐쇄됨
  • 결정 테이블(Decision Table) : 사용자 요구사항에 따라 원인(조건)과 결과(액션을) 결정 테이블에 정리하면 다음과 같다.
    결정 테이블의 테스트 케이스별로 각 조건에 따른 테스트 데이터를 준비하여 테스트를 수행한다.
테스트케이스 1 2 3 4 5 6
원인(조건) 자동화된 승차원 Y   N   Y  
자동화된 정액권   Y   N   Y
승차시 개찰되어 있음 Y Y N N N N
결과(액션) 자동 개찰구 통과 가능 X X        
역원에게 안내됨     X X    
개찰구 폐쇄됨         X X

예제 2

  • 사용자 요구사항
    OS종류가 Windows XP, 버전이 Home 이면서 라이센스가 유효하다면 자동 업데이트를 수행하고, OS가 Windows XP가 아니거나 버전이 Home이 아닌 경우에는 자동 업데이트를 수행하지 않는다.
    그리고OS가 Windows XP 버전이 Home이지만 라이센스가 유효하지 않다면 라이센스 발급유도 메시지를 발생한다.
  • 원인(조건)
    • Windows XP
    • Home
    • 라이센스 유효
  • 결과(액션)
    • 자동 업데이트 수행
    • Don’t Action
    • 라이센스 발급유도 메시지 발생
  • 결정 테이블(Decision Table): 사용자 요구사항에 따라 원인(조건)과 결과(액션을) 결정 테이블에 정리하면 다음과 같다.

결정 테이블 (Decision Table) 테스팅 기법 적용과 제약사항
  • 결정 테이블 테스팅(Decision Table Testing) 기법은 복잡한 비지니스 규칙을 구현해야 하거나 규칙이 조건의 조합으로 나타내어 질 수 있을 때, 조건들이 조건과 관련한 추상적인 액션을 가지고 있는 경우 사용가능하다.
  • 결정 테이블(Decision Tbale)은 시스템이 구현해야 하는 복잡한 비지니스 규칙을 문서화하는데 사용된다. 부가적으로, 결정 테이블은 테스트 케이스를 생성하는데 가이드를 제공한다.
  • 조건은 다양한 입력 조건을 나타내며, 액션은 입력 조건의 다양한 조합에 대응되어 실행되어야 하는 프로세스를 나타낸다. 각 룰은 그 룰과 연관된 실행 결과를 발생시키는 조건의 조합을 정의한다.
  • 각 룰(수직 컬럼)마다 최소한 하나의 테스트 케이스를 작성한다.
    룰이 이진 조건이라면 조합마다 하나의 테스트 케이스로 도출하며 값의 범위라면 값의 양 끝 최소, 최대 값 모두를 테스팅 값으로 고려하는 등 결정 테이블 테스팅 기법과 경계값 분석 테스팅 기법 또는 동등 분할 기법을 결합하여 사용한다.

2. 유스 케이스(Use Case)로부터의 테스트 케이스 도출 기법

사용자 시나리오(Use Case)를 실행하도록 테스트 케이스를 설계하는 블랙박스 테스트 설계 기법

적용방법
  1. 이벤트 흐름(Event Flow) 작성
  2. Use Case 시나리오 도출
  3. 테스트 케이스 도출
이벤트 흐름(Event Flow) 작성
  • 시스템 명세를 기반으로 Event Flow를 작성. 예를 들면, 아래 그림에서 기본 흐름(Basic Flow)과 선택 흐름(Alternative Flow)을 반영한 Use Case 내에 다양한 경로가 화살표로 표현되어 있다.
  • 검은 직선으로 표현된 기본 흐름은 Use Case를 관통하는 가장 단순한 경로이다.
  • 각 선택 흐름은 기본 흐름에서 시작하지만 특정한 조건에 따라 선택 흐름이 수행된다. 선택 흐름은 기본 흐름을 다시 만날 수(선택 흐름1과 3)도 있고, 또 다른 선택 흐름을 가지를 칠 수(선택 흐름 2)도 있으며, 기본 흐름을 다시 만나지 않고 Use Case를 종료할 수(선택 흐름 2와 4)도 있다.

‘계좌인출’ 유즈 케이스 명세서

기본 흐름 이 Use Case은 준비상태에 있는 ATM을 가지고 시작한다.
  • 예금인출 초기화 : 고객은 은행카드를 ATM 기계의 카드리더에 삽입
  • 은행카드 검증 : ATM은 카드에서 계좌를 읽어 사용 가능한 은행카드인지 체크[선택흐름-1]
  • 비밀번호 입력 : ATM은 고객에게 4자리 비밀번호를 요청
  • 계좌번호 & 비밀번호 검증 : 계좌번호가 유효한지 그리고 그 계좌번호에 대한 비밀번호가 유효한 것인지 확인[선택흐름-4]
  • ATM 거래 유형 선택 : ATM는 ATM에서 거래할 수 있는 다양한 유형을 보여준다. 이 흐름에서 고객은 항상 “예금 인출”을 선택함[선택흐름-2], [선택흐름-5]
  • 금액 입력 : 고객은 인출할 금액을 입력 이 흐름에서 고객은 미리 설정된 금액(10만원, 20만원, 40만원 또는 100만원)을 선택.[선택흐름-3]
  • 인증 : ATM은 트랜잭션으로 카드ID, 비밀번호, 금액, 계좌정보를 보냄으로써 은행시스템과의 확인 과정 수행 [선택흐름-6], [선택흐름-7]
  • 지급 : ATM은 고객에게 돈을 지급
  • 카드 반환 : 은행카드를 고객에게 반환
  • 영수증 : 영수증을 고객에게 출력 지급. ATM은 내부 Log 기록

  • Use Case 시나리오 각각에 대하여 Test Case를 정의한다.
    각 Use Case 시나리오에서 테스트 케이스를 도출하는 것은 특정 Use Case 시나리오가 실행되도록 하는 특별한 조건을 식별함으로써 이루어진다.
  • 위의 7개의 Use Case 시나리오 각각에 대하여 Test Case를 정의한다.
    테스트 케이스는 매트릭스(Matrix)나 결정 테이블(Decision Tables)을 이용해 식별할 수 있다.
    일반적인 형태인 매트릭스에서 각 줄(Row)은 각 테스트 케이스를 나타내고, 칼럼들은 Test Case속성을 나타낸다.

  • 매트릭스 개발을 위해서는 우선 Use Case 시나리오 수행에 필요한 데이터 식별에서부터 시작한다.
