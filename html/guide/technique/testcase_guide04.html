비기능 테스트를 위한 Test Case 도출

1. 개발표준 준수 여부를 체크하기 위한 공통 점검항목(Test Case) 도출

사용성, 유지보수성 등 비기능적 요구사항에 대한 테스트 방안으로 개발표준 준수여부를 검사하는 공통 점검항목(Test Case)를 도출할 필요가 있다.

개발 표준과 관련된 Test Case는 모든 프로그램이 공통으로 준수해야 할 사항으로 일반적인 형태의 Test Case처럼 대상 단위모듈별로 도출하지 않기 때문이다.

이러한 공통 점검항목(Test Case) 명세는 별도로 작성하여 테스트를 수행하는 담당자에게 배포하여 테스트 수행시 활용하도록 한다.

공통 점검항목(Test Case)이 테스트 되는 단계는 일반적으로 단위 테스트 단계이다.

공통 점검항목(Test Case)로 도출될 필요가 있는 내용은 다음과 같다.

  • 화면구성(Layout, Color, 크기, 폰트 등 화면 구성등의 속성)
  • 화면 I/F(업무처리에 따른 화면간 I/F, Paging, 에러/경고/도움말 메시지 처리 등)
  • 데이터 입력형식(문자, 숫자, 날짜 등의 입력 Format과 크기 그리고 필수 입력유무)
  • 데이터 출력형식(문자, 숫자, 날짜, 도표 등의 출력 Format)
  • 소스코드 코딩 규칙(주석처리, Naming rule, Indentation 등)

위의 사항은 대부분 개발 표준 가이드에 제시되는 내용으로 Test Case는 개발 표준 가이드를 참조하여 작성한다.

공통 Test Case는 아래와 같이 일반적인 Test Case와는 다른 형식으로 작성된다.

2. 보안/접근 테스트를 위한 Test Case 도출

유즈케이스 다이어그램(Use Case Diagram)에 기술되어있는 Actor와 Use Case는 시스템의 외부 사용자들과 시스템이 특별한 Actor에게 값을 넘겨주는 동작들 사이의 상호동작을 설명한다.

복잡한 시스템들은 많은 Actor들을 포함하고 있는데 이런 Actor들이 명시된 데로 Use Case를 수행하는지 확인할 수 있도록 Test Case를 개발해야 한다.

Actor의 유형에 따라 Use Case Flow of Event에 차이가 있다면 특히 이것은 보안/접근 테스트를 수행하는데 있어 중요한 사안인다.

예를 들어, ATM Use Case에서 Actor인 “고객”의 은행카드가 ATM를 소유한 은행의 카드인지 아니면 경쟁은행의 카드인지에 따라 다른 Use Case Flow of Event가 수행될 수 있을 것이다.

"Use Case로부터 기능 Test Case를 도출하는 기법"과 마찬가지로 보안/접근을위한 Test Case를 도출할 수 있다.

3. 구성 테스트(Configuration Testing)를 위한 Test Case 도출

일반적인 분산 시스템에서는 지원되는 H/W와 S/W의 많은 조합이 있을 수 있다.

테스트는 다른 OS, 브라우저, CPU 속도 등 다양한 만족할 만한 성능을 확인하기 위하여 수행될 수도 있다.

더 나아가 테스트는 다양한 구성요소의 상호작용에서 발생하는 결함을 제거하기 위해 구성 요소 조합에 대한 테스트를 포함할 필요가 있다.  

예를 들어, 어떤 프로그램의 인스톨에 의해 설치된 DLL의 버전이 다른 프로그램에 의해 사용되는 동일한 DLL의 출동하지 않는지 확인해야 한다.

구성 테스트를 위한 Test Case 도출 시 아래 지침을 활용할 수 있다.

구성간의 조합의 수가 많을 경우, Pairwise기법을 적용하여 최소화 할 수 있다.

  • 각 핵심 구성에 해당하는 적어도 하나의 Test Case가 있는지 확인.
    이것은 테스트 대상 시스템의 환경에 대한 요구된 H/W와 S/W 구성을 확인하고, 구성의 우선순위를 결정하며 가장 우선순위가 높은 구성들이 먼저 테스트 될 수 있도록 함
    • 프린터 지원
    • 네트워크 연결 - LAN/WAN
    • 서버구성 – 서버 드라이버, 서버 하드웨어
    • 데스크 탑이나 서버에 인스톨된 소프트웨어
    • 모든 인스톨된 소프트웨어에 대한 버전
  • 문제가 있을 것 같은 각 구성에 대하여 적어도 하나의 Test Case를 작성
    • 최하의 성능을 가진 H/W
    • 호환성에 문제가 있었던적이 있는 S/W 인스톨
    • 가장 느린 LAN/WAN 연결 상에서 클라이언트가 서버를 엑세스
    • 불충분한 자원(느린 CPU속도, 최소한의 메모리나 해상도, 디스크 스페이스 등)

4. 설치 테스트(Installation Testing)를 위한 Test Case 도출

인스톨레이션 테스트는 테스트 대상이 모든 가능한 인스톨 시나리오 상에서 정상적으로 수행하는지를 검증하는 것이다.

테스트 대상에 처음으로 인스톨 하는 경우도 있을 것이고, 예전에 인스톨되었던 곳에 다시 하는 경우도 있을 것이고, 새로운 버전을 인스톨 하는 경우도 있을 것이다.

인스톨 테스트는 불충분한 디스크 공간과 같이 비정상적인 조건에 직면했을 때 테스트 대상이 만족하게 수행하는지 또한 확인되어야 한다.

Test Case는 아래와 같은 소프트에어에 대한 인스톨 시나리오를 포함해야 한다. 삭제 테스트를 위한 Test Case 도 유사하게 도출될 수 있다.

  • 배급 매체 (디스크, CD-ROM, 파일 서버 등)
  • 신규 인스톨
  • 전체 인스톨
  • 사용자 정의 인스톨
  • 업그레이드 인스톨

Host-Based System과 다르게 일반적으로 인스톨 프로그램은 서버와 클라이언트 프로그램으로 구분될 수 있다. 따라서 Clinet, Middle-Tier, Server를 포함한 테스트 대상의 모든 컴포넌트에 대한 인스톨레이션 테스트를 수행하는 것이 중요하다.

다른 비기능 테스트를 위한 Test Case 도출

이상적으론 Use Case 모델, 디자인 모델, 보충명세서(Supplementary)에서 Test Case를 도출하는데 필요한 모든 정보를 찾아야 한다.

그러나 이점에서 그것들에서 찾은 것들을 완전하게 할 필요가 있다는 것이 비정상적이지는 않다.

예

  • 운영 테스트에 대한 Test Case
  • 성능 병목 조사, 시스템의 처리 능력, 테스트 대상의 스트레스에 대한 Test Case

대부분의 경우에 지금까지 도출한 Test Case를 분류하거나 통합하면서 이런 Test Case를 찾을 수 있다.
