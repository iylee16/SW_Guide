Test Case 도출 기법

아래 표는 테스트 단계별 테스트 유형을 예를 들어서 설명한 것이다.

일반적으로 Test Case 도출을 하는 방법은 단위 테스트, 결합 테스트, 시스템 테스트 등과 같이 테스트 단계별로 구분되기 보다는 수행되어야 하는 테스트 유형에 따라 달라진다.

예를 들어 접근권한(보안)과 관련된 테스트의 경우, 단위 테스트에서 수행되어야 하는 Test Case도 있으며 통합테스트에서 수행되어야 하는 Test Case도 존재할 수 있다.

테스트 단계 테스트 유형 설명
단위테스트 기능 개별 Component (OO 개발 방법론) 기능 테스트
단위 업무 기능 테스트
표준준수 표준(화면/입출력형식/프로톨콜/코딩 룰등) 준수여부 체크
보안 단위 프로그램내 사용자별 기능 제한 사항 체크
성능 대상 단위업무에 대한 개별 성능 측정
결합 테스트 Component통합 Component 통합 하에서 기능(OO 개발 방법론) 테스트 : 단위테스트에서 수행되기도 함
데이터 연계 업무 처리 흐름에 따른 데이터 흐름 테스트
시스템간 통합 타 시스템과의 프로세스I/F 테스트
타 시스템간의 데이터 I/F 테스트
소프트웨어 통합 가능한 소프트웨어 구성에 따른 시스템의 운영여부 테스트
하드웨어 통합 시스템 구성 하드웨어간의 I/F 테스트
보안 사용자별 접근 제한(통합환경)
시스템 기능 단위 테스트와 통합과정에서 이루어졌던 기능 중 시스템 전반에 걸친 주요 기능에 대해 테스트
시스템테스트 성능 사용자 수 대비 응답시간 측정
볼륨 시간당 대상 업무에서 처리될 수 있는 최대 건수 측정
스트레스 시스템의 한계사항 하에서의 부족한 리소스 및 시스템 상태 확인
구성 소프트웨어들의 구성, 소프트웨어 환경설정, 하드웨어의 물리적 구성 등에서 최적의 운영환경 발견하기 위한 테스트
복구 시스템 Down, DB Failure 등의 사항에서 정상적인 가동에 가정과 시간에 대한 테스트
보안 시스템 접근 제한에 대한 테스트
설치 시스템의 설치 및 인스톨 프로그램의 설치 테스트

테스트 케이스는 테스트나 관련 요구사항 유형에 의해 분류하므로 다양한 Test Case가 도출될 수 있다. 보통 적용하는 방식은 테스트할 각 요구사항에 대하여 최소한 두개 이상의 Test Case를 개발하는 것이다.

  • 요구사항이 반영되었는지 확인하는 Test Case (Postive Test Case)
  • 비정상적이거나 기대하지 못한 조건과 데이터를 반영하여 요구사항이 기대 상태 하에서 이루어졌음을 확인하는 Test Case (Negative Test Case)

(기능) 단위/결합 테스트를 위한 Test Case 도출

1. 동등분할 기법(Equivalence Partitioning)

동등 분할된 영역에서 대표값으로 수행하도록 테스트 케이스를 설계하는 블랙박스 테스트 설계 기법. 원칙적으로 테스트 케이스는 각 영역을 적어도 한번은 커버할 수 있게 설계한다.

입력 데이터의 영역(Domain)을 유사한 특징을 가진 그룹 (클래스)로 분류하여 각 클래스에서 대표 테스트케이스를 도출한다.

무한한 수의 가능한 테스트 케이스를 훨씬 수가 적지만 같은 효과를 가지도록 테스트 데이터를 분류하는 방법이다.

동등 분할 기법 적용방법
  • 주어진 시스템 정보(예, 시스템 명세서)를 분석하여 입력 도메인을 유사한 특징을 가진 동등 클래스 (Equivalence Class)로 분할
  • 분할된 클래스에서 각 클래스를 대표하는 테스트 케이스 도출
동등 클래스(Equivalence Class)
  • 동등 클래스(Equivalence Class)는 테스트 입력값의 부분 집합으로, 각 Class내의 임의의 값이 그 Class를 대표할 수 있어야 한다.
    즉, 만일 Class 내의 한 값을 테스트 케이스로 사용하여 결함을 발견한다면, 다른 값들을 넣었을 때도 결함을 발견할 수 있어야 한다는 것을 의미함.
  • 2가지 종류의 동등 클래스가 존재하는데, 하나는 유효한 입력조건을 대표하는 Valid Class이고 그 외의 다른 경우를 Invalid class라고 함
  • Equivalence Partitioning은 2단계로 이루어지며, 1단계는 Equivalence Class 작성이고, 2단계는 테스트케이스 작성이다.
동등 클래스(Equivalence Class) 작성
  • 만일 입력 조건이 “범위”라면 (예, 물품수의 범위는 1~999사이여야 함), 하나의 Valid class (1< 물품수<999)와 2개의 Invalid class (물품수<1, 물품수>999)를 정의
  • 만일 입력조건이 “여러 개의 값들”이라면(예, 1개에서 6개까지의 물품이 입력됨) 1개의 Valid class와 2개의 Invalid class(아무것도 표시되지 않은 경우, 7개 이상 표시된 경우)를 정의
  • 만일 입력조건이 “특정 값의 집합”이고, 각 값들이 별도로 취급되는 경우(예, 버스, 트럭, 택시, 보행자, 자전거), 각 값마다 한개씩의 Valid class와 하나의 Invalid class(예, 트레일러)를 정의
  • 만일 입력조건이 “must be” 인 경우 (예, id의 첫글자는 영문자이어야 한다), 하나의 Valid Class(영문자인 경우)와 하나의 Invalid Class(영문자 아닌 경우)를 정의
  • 만일 Class내의 항목들이 동일한 방식으로 처리되지 않는 경우, 보다 작은 class로 세분화.
테스트 케이스(Test Case) 도출
  • 각가의 동등 클래스(Equivalence class) 일련 번호를 부여
  • 모든 Valid class가 커버될 때까지 테스트 케이스를 추가(즉, 하나의 케이스가 여러 Class를 커버할 수 있음)
  • 각 Invalid class마다 하나의 테스트 케이스를 작성
  • Invalid Class마다 케이스를 작성하는 이유는, 하나의 입력 오류가 다른 입력오류를 체크하지 못하도록 하기 때문이다.
    예를 들어, “종류는 (HARD, SOFT, LOOSE)중 하나이고 수량은 (1-9999)”일때 테스트 케이스에서 사용한 입력 Domain이 (TIGHT, 0)인 경우 “종류”에 따른 에러 때문에 “수량”에 대한 입력을 검증하지 않는 경우가 발생한다.

예제 1

프로그램 설계 사양서에 “시작일은1990-1-1이후이며 종료일은 시작일보다 빠를 수 없다.”라고 기재된 경우
(일자 입력의 정확성은 프로그램 내부적으로 처리하고 있다고 가정, 즉 없는 날짜인 2월30일 등의 입력은 사전 처리되어짐)

입력조건 Valid Classes Invalid Classes
시작일 (1) 시작일 >= 1990-1-1 (3) 시작일 = 공란
(4) 시작일 <1990-1-1
종료일 (2) 종료일 >= 시작일 (5) 종료일 = 공란
(6) 종료일 < 시작일

1 1~100 1 ~ 100 2.1.2                           0 이하의 값

2. 경계값 분석(Boundary Value Analysis)

경계값에 기반하여 테스트 케이스를 설계하는 블랙박스 테스트 설계 기법.

즉, 경계값 분석은 동등 분할(equivalence partition)된 경계의 유효한 값과 경계에서 가장 가까운 유효하지 않은 값을 테스트 데이터로 선택하여 컴포넌트나 시스템을 테스트하는 기법이다.

개발자가 분할된 경계값을 처리하면서 개발상의 실수 가능성이 높다는 것에 착안하여 테스트 데이터를 선정한다. 결국 경계값 분석은 등가 분할 기법(equivalence partitioning)을 확장하여 등가 분할 내에 있는 임의의 값이 아닌 분할된 영역의 경계에 있는 값을 테스트 데이터로 이용하여 보다 많은 결함을 발견하는 테스트 케이스(test case)를 작성하는 기법이다.

  • Equivalence Partitioning의 한 변형된 형태의 기법.
  • Equivalence class 내부 값이 아닌 경계 값을 주관심 대상으로 한다.
Equivalence partitioning과의 차이점
  • Equivalence class내의 임의의 값을 선택하는 대신, 각 class의 경계값이 선택되도록 함
  • 입력조건만이 아닌 출력조건 또한 고려함
경계값 분석 적용방법
  1. 입력조건이 “범위”인 경우, 범위의 양끝을 테스트 케이스로 작성하고, 범위를 바로 넘어선 값을 Invalid 테스트 케이스로 작성

    ex) 유효한 범위가 “-1.0 ~ +1.0”인 경우, -1.0, 1.0, -1.001, 1.001을 테스트 케이스로 작성

  2. 입력조건이 “몇 개의 수”인 경우, 가장 적은 개수와 가장 많은 개수 및 이 개수 이전과 이후를 테스트 케이스로 작성

    ex) 입력 파일인 1~255개의 레코드를 포함해야 하는 경우, 0,1,255,256개의 레코드를 가진 테스트 케이스 작성

  3. 출력 조건에 대하여 ①번 기준을 적용

    ex) 예를 들어, 출력값이 0~2000 사이여야 하는 경우, 출력이 0,2000, -1, 2001이 될 수 있는 입력값을 계산하여 테스트 케이스로 작성

  4. 출력조건에 대하여 ②번 기준을 적용.

    ex) 4개 이하의 값들을 출력해야 하는 경우, 출력값의 수가 0,1,4,5가 될 수 있는 입력값을 테스트케이스로 작성

  5. 입력조건 또는 출력조건이 순차 집합인 경우(예를 들어, 순차 파일, 순차리스트, 테이블), 최초 및 마지막 항목에 주의를 두어 테스트 케이스 식별
