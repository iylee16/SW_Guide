개요

1. 목적

본 문서는 프로젝트팀에서 테스트 전략을 수립할 때 필요한 지침으로 테스트 전략 활동으롤 통해 마스터 테스트 계획서(Master Test Plan)을 작성하는 가이드라인을 제공한다.

2. 적용 범위

본 문서에서 기술하고 있는 테스트 전략 활동은 마스터 테스트 계획서 작성을 대상으로 한다.

※ 특정 테스팅 레벨의 상세한 테스트 계획 수립 가이드는 해당 테스트 레벨 수행 가이드라인을 참고한다.

3. 테스트 전략 활동 정의

테스트 전략 활동이란 개발 프로젝트 및 시스템의 특성을 고려하여 적합한 테스트 목표 및 실행 전략, 테스트 범위(scope), 접근 방법(approach), 테스트 수행시 요구되는 자원 (툴 포함), 테스트 활동 및 절차, 테스트 수행 조직(Staffing), 테스트 일정(Schedule), 구성원의 교육요건등을 사전에 정의하여 프로젝트 수행 시 원활한 테스트 수행을 위한 사전 활동이다.

테스트 전략 수립은 성공적인 테스트 수행 여부를 결정짓는 중요한 요소중 하나이나 개발 일정상의 제약, 테스트 전문 인력의 부족, 테스트 계획의 중요성에 대한 인식 부족 등으로 종종 생략된다.
그러나 전체적인 테스트 계획(Master Test Plan) 없이 대규모의 테스트를 수행하는 것은 대규모의 개발 프로젝트를 전체적인 계획없이 시스템 개발을 추진하는 것과 유사하므로, 테스트 전략 활동은 성공적인 테스트 수행을 위해서 반드시 필요한 과정이다.

테스트 전략 지침

1. 테스트 전략 주요 TASK

일반적으로 테스트 전략 활동은 SW개발 프로세스의 요구사항 분석단계 말에서 이루어지며 사용자 요구사항이 파악되는 시점에서 개발시스템의 Risk, 범위, 담당자 지정, 확보된 예산 등을 고려하여 향후 수행되어질 필요가 있는 테스트에 대한 전체 청사진(BluePrint)을 구상하는 단계라고 볼 수 있으며 이러한 활동의 결과로 마스터 테스트 계획서를 작성한다.

2. 테스트 요소(Test Factor) 식별

시스템과 관련된 위험을 평가하고, 그 위험과 관련된 테스트 요소를 정의한다.

테스트 수행 시 제거되어야 하는 시스템 관련 위험이나 이슈를 테스트 요소라 하며 테스트 요소 자체가 위험은 아니지만, 비즈니스 위험으로 변환될 수 있기 때문이다.
예를 들어, 소프트웨어가 사용하기 쉽지 않다면 처리 결과가 부정확할 수 있다.

테스트 요소를 식별하기 위해서는 다음과 같은 활동들이 요구된다.

  1. 테스트 요소 식별
    • (테스트 대상) 시스템 및 관련 비즈니스 위험을 식별
    • 테스트 수행 중에 평가되어야 하는 위험을 기반으로 테스트 요소를 식별
    • 심각도 및 위험도를 고려하여 테스트 요소를 분류
  2. 식별된 테스트 요소에 대한 테스트 종류별 테스트 활동 정의
    • 개발프로젝트 생명주기(Lifecycle)을 파악
    • 테스트 요소와 관련한 위험을 최소화하기 위한, 개발프로젝트의 생명주기 각 단계에 적합한 테스트 종류를 정의
    • 테스트 종류별 테스트 방법을 정의

예를 들어, ‘트랜잭션의 정확성’은 심각한 비즈니스 위험이며, ‘정확성’은 소프트웨어 개발 라이프사이클에 따라 추적되어야 할 테스트 요소이다.

테스트 전문팀 혹은 프로젝트팀은 시스템을 통해 입력/처리/출력되는 데이터가 프로젝트 라이프사이클 동안 정확하고 완전하다는 것을 보장해야 한다.

아래는 테스트 전략 수립 시 고려해야 할 테스트 요소의 예이다.

시스템 위험 테스트 요소 근 거 단위 테스트 통합 테스트 시스템 테스트
컴퓨터 조작의 부정확성 정확성 시스템에서 수행되는 중요한 조작의 정확성 확인 정확성을 위한 처리 과정 테스트 중요한 단위(Unit) 사이의 데이터 흐름과 전달 테스트 완전한 시스템에서 정확성과 데이터 흐름 테스트
데이터 무결성 /마스터 데이터 유지 파일 무결성 파일과 데이터베이스 컨텐츠의 정확성 확인 데이터베이스 갱신의 정확성 테스트 트랜잭션 완료 시, 마스터와 트랜잭션 기록의 정확성 테스트 완전한 시스템에서 마스터와 트랜잭션 기록의 정확성 테스트
보안 권한 부여 사용자의 처리 권한 부여 확인 N/A 다른 사용자를 이용한 처리 권한 테스트 다른 사용자를 이용한 처리 권한 테스트
시스템 실패 프로세스의 연속성 시스템의 복구 성능 확인 N/A N/A 시스템 기능 복구 테스트

테스트 요소는 다양한 소프트웨어 개발 라이프사이클 단계에서의 테스트 액티비티 범위를 정의하는 중요한 항목이다.

테스트 범위는 시스템의 중대성뿐만 아니라 프로젝트의 일정 제약 사항에도 영향을 받는다. 테스트 범위는 위에서 언급된, 그리고 테스트 할 필요가 있는 테스트 요소를 포함한다.

향후 각 단계별 상세 테스트 계획과 설계는 테스트 전략 수립 단계에서 식별된 테스트 요소에 영향을 받는다.

3. 테스트 범위(Test Scope) 정의

프로젝트 범위 및 상위 요구사항을 분석하여 테스트 범위를 설정한다.

테스트를 통하여 적합성을 검증 받아야 할 테스트 범위(Scope)는 대상 시스템 전체가 될 수도 있고, 경우에 따라 일부분이 될 수도 있다.

선정된 테스트 범위는 향후 테스트 단계별 상세 계획 수립 시, 요구분석 및 설계 단계에서 정의된 각 테스트 요구사항 들을 검토하여 테스트 항목(Test Items), 테스트할 항목 특성(Features to be tested)과 테스트하지 않을 항목 특성(Features not to be tested)을 명세화 하는 기반 자료로 활용된다.

4. 테스트 단계별 접근방법(단계 및 수행방안) 정의

  1. 개발 생명주기별 테스트 단계 정의

    테스트 단계는 시스템 개발 생명주기별로 어떤 테스트를 수행할 것인지를 나타낸다.

    마스터 테스트 계획에서 결정해야 할 중요한 사항중의 하나가 프로젝트에서 적용할 테스트 단계와 단계별 수행방안을 결정하는 것이다.

    아래 표는 일반적인 테스트 단계를 나타낸다.

    테스트 단계 (TEST LEVEL) 설명
    단위 테스트 단위 모듈의 기능, 성능, 기타 품질속성(Quality Attribute)을 테스트
    통합 테스트 통합된 모듈간의 내∙외부 I/F 및 기능을 통합이 진행됨에 따라 반복적으로 테스트
    시스템 테스트 시스템의 전체적인 기능을 테스트
    부하상태에서 시스템 성능 측정 (Performance/Load 테스트)
    회귀 테스트 결함 수정이나 변경사항 반영 후 테스트를 반복하여 시스템 정합성 재확인
    인수/운영 테스트 시스템이 사용자의 인수기준을 만족하는지 확인

    모든 프로젝트와 모든 릴리즈에 표준적인 테스트 단계를 사용할 수도 있으나 테스트 할 시스템의 크기와 복잡도, 시스템의 중요도, 개발프로세스 성숙도, 고객 요청사항, 테스트 일정, 테스트 환경, 테스트 참여 인원 및 조직과 경험, 위험 등을 고려하여 개발프로젝트에서 수행할 테스트 단계를 결합, 생략, 세분화하여서 수행할 수 있다.

    • 언제 테스트 실행이 시작되는가?
    • 얼마나 많은 테스트 공간이 사용될 것인가?
    • 파일럿을 수행할 것인가? (즉, 단일 또는 제안된 수의 장소에서 실행되어지는 테스트)
    • 어떤 테스팅 기법을 활용할 것인가? (예를 들면, 현업과의 "buddy" 테스팅, 인스펙션, 워크스루 등)
    • 어떤 테스트 엔지니어 및 참여자가 테스트 계획 수립, 설계, 실행에 요구되는가?
    • 어떤 테스트 단계(예를 들면, 단위, 결합, 시스템, 운영 테스트 등)이 사용될 것인가?

    위 표에 명시된 테스트는 프로그램 실행을 통하여 결과를 검증하는 동적 테스트(Dynamic Test) 활동들인데, 이 외에도 개발 산출물의 정확성, 완전성, 일관성 등을 검토하는 검증(Verfication) 활동도 필요시 테스트 영역으로 포함시키고 테스트 단계의 일환으로 테스트 계획서상에 포함시킬 수 있다.

    이러한 테스트 단계 정의 예는 아래와 같다.

    개발 단계(Phase) 테스트
    요구사항 분석 요구사항 테스트 (분석 품질검토회)
    설계 설계 테스트 (설계 품질검토회)
    제작 코드 인스펙션(Code Inspection), 단위 테스트
    테스트 결합 테스트, 시스템 테스트,
    유지보수 회귀 테스트(Regression-Test)
  2. 테스트 수행방안 정의

    시스템 특성이나 개발 특성에 따라 각 테스트 단계별로 특정 유형의 테스트가 필요하기도 하고 불필요하기도 하다.

    테스트 전략 수립에서는 프로젝트에서 수행될 각 테스트 단계별로 필요한 테스트를 유형을 결정하고 어떻게 테스트를 수행할 것인인지에 대한 내용을 개략적으로 서술한다. 아래 표는 프로젝트에서 적용가능한 일반적인 테스트 유형의 예를 보여준다.

    테스트 유형 (Test Type) 설명
    기능 테스트 요구사항 대비 기능이 모두 구현되었는지 완전하고 정확하게 구현되었는지 확인하는 테스트
    성능 테스트 제한된 시스템 자원 하에서 시스템의 성능을 측정하는 테스트
    신뢰성 테스트 옳고 일관된 결과를 얻기 위해 요구된 기능을 수행할 수 있는 정도를 테스트
    사용성 테스트 사용자의 사용편의성을 테스트
    유지보수성테스트 시스템의 유지보수가 얼마나 용이한가 테스트
    효율성 테스트 시스템의 자원효율성과 시간효율성을 테스트
    테스트 접근방법의 일반적인 유형

    테스트 접근방법에는 상향식 혹은 하향식이 있으며, 테스트 전략 수립 시 프로젝트 테스트 성격에 따라 적절하게 선택하여야 한다.

    크고 복잡한 시스템 개발 시에는 상향식 접근 방법이 용이하다. 아래는 각 접근 방법의 장단점을 기술한 것이다.

    • 하향식 접근방법

      최상위 단위 프로그램이 스텁으로 대체된 하위 단위 프로그램을 호출함으로써 테스트하는 기법으로, 최하위 단위 프로그램이 테스트될 때까지 테스트가 진행된다.

      하향식 전략은 단위 프로그램 구조상 하위 단위 프로그램의 복잡도나 변경에 따라 영향을 받기 때문에 많은 비용이 소요되며, 단위 테스트에는 적합하지 않다.

      그러나 각 단위 프로그램에 대한 테스트가 완료되었다면, 단위 프로그램의 통합을 위해 하향식 접근 방법을 사용할 수는 있다.

      • 하향식 접근방법의 장점
        • 상세 설계 및 코딩과 동시에 테스트를 실행할 수 있으므로 개발 기간이 단축된다.
        • 일반적인 구조적 설계에서 구조의 상위 단위 프로그램은 기본 기능을 제공하고, 하위 단위 프로그램은 상세 내역을 구현하며, 하향식 단위 테스트를 통해 명확한 기능에 대한 조기 통합을 할 수 있다.
          이를 통해 단위 테스트에 요구사항 기반 접근방법을 적용할 수 있다.
        • 하향식 단위 테스트에서는 하위 단위 프로그램을 테스트할 경로의 유무를 판단함으로써 하위 수준의 단위 프로그램에 대한 중복 기능을 식별해 낼 수 있다.
      • 하향식 접근방법의 단점
        • 하향식 단위 테스트는 스텁에 의해 제어되며, 테스트 케이스는 종종 여러 스텁을 필요로 한다. 따라서 테스트가 더 복잡해지고 결과적으로 개발/유지보수 비용이 더 소요된다.
        • 에러 처리 코드와 같은 하위 기능 테스트에는 실용적이지 못하다.
    • 상향식 접근방법

      자신을 호출하는 단위 프로그램과는 별개로 테스트되는 것으로, 최하위 단위 프로그램이 먼저 테스트되고 상위 단위 프로그램이 테스트된다.

      상향식 접근방법은 단위 프로그램 구조상 하위 단위 프로그램들의 복잡도나 변경에 영향을 받기 때문에 많은 비용이 소요되며, 일정이 촉박한 프로젝트에는 적합하지 않다.

      • 상향식 접근방법의 장점
        • 상향식 접근방법은 상대적으로 간단히 하위 레벨의 단위 테스트 할 수 있다.
        • 상향식 테스트에 사용되는 테스트 케이스는 구조적 설계정보 없이 단지 기능 설계 정보를 이용해서 설계할 수 있다.
          따라서, 세부 설계 명세서에 구조적 상세 정보가 부족하여도 단위 테스트까지 상향식 방법을 적용할 수 있다.
        • 상향식 단위 테스트는 초기에 하위 기능들을 통합할 수 있으며, 테스트가 진행되는 동안 상위 기능들이 추가로 통합된다.
      • 상향식 접근방법의 단점
        • 테스트가 진행됨에 따라, 상향식 단위 테스트는 더 복잡해지고 결과적으로 개발이나 유지보수 시에 더 많은 비용이 소요된다.

5. 품질 목표 수준

테스트 단계에서 수집/관리가 필요한 품질 지표를 기록한다.

대부분 품질 목표 수준은 개발계획서’2. 프로젝트 성과 목표’에 기재하도록 되어 있으며 테스트 관리자(또는 PM, 테스트 설계자)는 테스트 단계에서 필요한 데이터를 수집하여 측정지표로 관리하여야 한다.

Appendix. 용어정의

본 영역에서 기술되지 않은 용어는 POSCO SM 표준 테스트 프로세스내 용어 정의를 따른다.

테스트 전략 (Test Strategy)
하나의 프로그램에 대해 어떤 테스트 레벨(test levels)을 수행할 것인지, 해당 테스트 레벨 내에서 어떻게 테스팅 할 것인지를 정의하는 상위 수준 문서.
테스트 요소 (Test Factor)
테스트 전략의 일부로 프로젝트와 관련된 위험이나 이슈를 말하며, 테스트 단계에서 제거하거나 완화시켜야 하는 시스템 위험이다.
테스트 시작 조건 (Test Exit Criteria)
정의된 (다음 단계) 작업의 진행을 허가하는 일반적이고 특정한 조건의 집합. 시작 조건의 목적은 시작 조건을 만족하지 못한 채 테스트 되어 추후에 더 많은 (헛된)노력이 소요되는 태스크(task)를 시작하지 못하도록 하는 것이다
테스트 종료 조건 (Test Exit Criteria)
절차가 공식적으로 완료되었음을 허가할 목적으로, 이해관계자들이 합의한 일반적이고 특정한 조건의 집합. 완료조건의 목적은 아직 종료하지 못한 미해결의 일부 업무가 여전히 존재 함에도 완료한 것으로 간주하려는 것을 방지하는 것이다. 완료조건은 언제 테스팅을 중지할 것인지에 대해 보고하고 계획하기 위해 사용된다. 테스트 중단을 결정하는데 필요한 전제 조건이다.
테스트 레벨 (Test Level)
함께 편성되고 관리되는 테스트 활동의 그룹. 테스트 레벨은 프로젝트에서 책임과 연관 되어있다. 테스트 레벨의 예로는 단위 테스트, 통합 테스트, 시스템 테스트 그리고 인수 테스트가 있다.
테스트 접근방법 (Test Approach)
특정한 프로젝트를 위해 테스트 전략을 구현하는 것. 테스트 접근법은 일반적으로 프로젝트의 목표와 리스크 평가 수행 결과를 기반으로 여러가지 테스팅 관련 의사결정을 한다. 이 밖에도테스트 접근법은 테스트 프로세스를 시작하는 시점, 적용할 테스트 설계 기법, 완료 조건과 수행 할 테스트 유형을 결정하는 역할을 한다.
테스트 유형 (Test Type)
하나 이상의 상호 연관된 품질속성에 대해 컴포넌트나 시스템을 테스팅 하고자 목표하는 테스트 활동 묶음. 테스트 유형은 신뢰성 테스트, 사용성 테스트, 회귀 테스트 등과 같은 특정 테스트 목표에 집중한다. 그리고 테스트 유형은 하나 이상의 테스트 레벨 또는 테스트 단계에서 발생할 수 있다
테스트 요구사항 (Test Requirement)
하나 또는 그 이상의 테스트 케이스에 의해 검증될 수 있는 컴포넌트나 시스템의 항목 또는 이벤트. 테스트 상황의 예는 기능, 트랜잭션, 품질 속성, 또는 구조적 요소 등이다.
품질 속성 (Quality Attribute)
항목의 품질에 영향을 미치는 특징(feature)이나 특성(characteristic)
개발 생명주기 (Development LifeCycle)
소프트웨어 제품의 개념 형성에서 시작하여 운영/유지 보수에 이르기까지 변화하는 개발의 전 과정
동적 테스팅 (Dynamic Testing)
컴포넌트나 시스템 소프트웨어를 실행하면서 수행하는 테스팅
정적 테스트 (Static Testing)
리뷰 또는 정적 코드 분석과 같이 소프트웨어의 실행 없이 명세 또는 구현, 개발 단계에서 컴포넌트나 시스템을 테스팅 하는 것.
