요구사항 명세

1. 요구사항 명세 활동

솔루션이 어떻게(How) 수행될 것인가 보다는 무엇(What)을 수행할 것인가에 초점을 두고 고객 또는 사용자의 요구들을 솔루션 개발에 있어 동일하게 이해하게끔 하는 것

요구사항 명세서란(Requirement Specification)

  • 요구사항에 대한 공식 문서
  • 참여자들 사이의 의사소통 수단 제공
  • 사용자, 분석가, 개발자 및 테스터들에게 공동의 목표 제시
  • 자연어나 정형언어 또는 그래픽 형태로 기술
  • 명확한 이해와 수행할 기능의 완전한 기술
  • 관점에 대한 다른 표현 사이에 일치성과 상호참조 확인
  • 생명주기의 다음 단계 자료로 사용
  • 설계, 검증 또는 테스트 계획 수립을 위한 기초 토대 마련
설계 요구사항

Solution

목표달성을 위한 해결방법 기술

“어떻게 수행될 것인가”

추상화

- 시스템이 이루어야 할 목표 기술

“무엇을 수행할 것인가”

HOW it is to be provided

“기술적 요소를 포함하는가?”

“특정구조나 알고리즘을 기술하는가?”

WHAT is needed

“해당 요구사항이 왜 필요한가?
  • 시스템은 비행성능 계측기를 포함해야 한다.
  • 시스템은 두 개의 백업시스템을 구성해야 한다.
  • 사용자는 시스템에 접근하기 위하여 패스워드를 사용하여야 한다.
  • 시스템은 비행성능을 측정해야 한다.
  • 시스템은 고장이 발생할 경우 운영 요구사항을 만족하여야 한다.
  • 시스템은 인증된 사용자만 접근을 허용하여야 한다.

요구사항 명세 절차

기능 요구사항 명세
  • 분석된 기능 요구사항의 지속적 관리를 위해 요구사항 명세기준에 따라 명세한다.
    • 요구사항을 지속적이고 효과적으로 관리하기 위해 표준화된 서식에 맞추어 요구사항 상세 내역을 작성한다.
    • 요구사항 명세 품질점검 및 보완
비기능 요구사항 명세
  • 분석된 비 기능 요구사항의 지속적 관리를 위해 요구사항 명세기준에 따라 명세한다.
    • 요구사항을 지속적이고 효과적으로 관리하기 위해 표준화된 서식에 맞추어 요구사항 상세 내역을 작성한다.
    • 요구사항 명세 품질점검 및 보완

2. 요구사항 명세 가이드

어떻게 작성해야 하나

  1. 고객 관점의 명세

    • 고객과 개발자가 이해하기 쉽게 작성한다
      • SRS의 Reader가 누구인가?
      • 그들에게 적합한 표현방식은 무엇인가?
    • 최신유행을 피한다 전문가의 자기중심적 특정 기법
      • 이해하기 쉽고 배울 필요가 없는 표현기법을 사용 : 자연어, Diagram
      • 조직의 실제와 요구사항작성자/독자가 사용하는 표현에 의존

  2. 고객 관점의 명세 → 시나리오 기반 요구사항 기술

    • 유즈케이스 기술서 (Use Case Description)

  3. 읽고 이해하기 쉬운 표현 사용

    • 긍정적인 표현을 사용한다.
      • 부정적인 요구사항은 긍정의 의미로 표현한다.
      • 다중 부정표현을 피한다.
    • 능동적인 표현을 한다.
    • 요구사항에 충분한 정보를 제공한다.
    • Model과 Picture를 포함한다.

  4. 명확한 표현을 사용

    • 모호한, 다중의미를 가지는 용어를 피한다.
      • usually, generally, often, normally, typically, user friendly, efficient, flexible, maximize, minimize, improved, easy, simple
    • 일관성 있는 용어를 사용한다.
      • Glossary 기술: 조직, 도메인의 표준용어사용
    • 요구사항범위를 명확하게 한다.
      • A/B : "A 그리고B", "A 또는 B", etc (... 등 등)
      • Support vs. provide

어느정도 상세하게 작성해야 하는가

상세함이 덜 요구된다

상세함이 더욱 요구된다
  • 고객들이 광범위하게 포함된다
  • 개발자들은 상당한 분야의 경험을 가지고 있다
  • 선례들을 사용할 수 있다
  • 패키지 솔루션이 사용될 것이다
  • 개발은 아웃소싱 될 것이다
  • 프로젝트 팀 멤버들은 지역적으로 흩어져 있다
  • 테스트는 요구사항을 근거로 한다
  • 정확한 평가가 필요하다
  • 요구사항 추적성이 필요하다

좋은 요구사항(well defined) 명세속성

명세속성

설명

1. 정확성 (Correctness)
  • 명세서 안에 기록된 모든 요구사항이 SW에서 만족됨

2. 명확성 (Unambiguous)
  • 명세서 안에 기록된 모든 요구사항이 한가지 의미로 해석됨
  • 솔루션의 각 특성들은 유일한 용어로 표현되고 그 용어는 한가지 의미를 가지고 있어야 함

3. 완전성 (Completeness)
  • 솔루션에서 중요한 요구사항들 특히 기능, 성능, 설계 제약사항, 외부 인터페이스 등의 요구사항이 모두 명세서에 포함됨
  • 구현 가능한 특정 상황에서 입력 데이터에 대한 SW의 반응에 대한 정의, 입력이 의미가 없는 것일지라도 입력에 대한 모든 반응이 기술됨

4. 일관성 (Consistency)
  • 명세서가 상위수준 또는 연관된 문서와의 레벨이 맞음 (내부적 내용)

5. 특이성 (Peculiarity)
  • 각 요구사항에 대한 중요성/우선순위를 표시해주고 특정기준에 따라 판단하여 중요성/우선수기를 표기하였음

6. 검토가능 (Verifiability)
  • 요구사항이 검증 가능해야 함

7. 수정용이성 (Modifiability)
  • 요구사항이 그 구조와 형태를 그대로 존속하면서 요구사항의 변경을 쉽고, 완전하고 일관성 있게 수행할 수 있는 것을 말함

8. 추적성(Traceability)
  • 각 원본이 명확하고 향후 개발이나 문서의 개선이 필요한 경우 참조할 수 있어야 함

9. 이해가능성(Understandability)
  • 요구 사항 산출물에 기술한 문장이 표준 형식을 따르며 적절한 문법을 따르고 있으며 다중문장을 배제하여 용이하게 이해 가능한지 여부

요구사항 구조(구문)

피해야 할 단어(용어)

  • 최상급/비교급 (가장좋은, 최고의, 보다 높은 품질, 보다 나은 기능 등)
  • 주관적 용어 (사용하기 쉬운, 효과적인 등)
  • 모호한 대명사 (그것, 저것, 이것 등)
  • 모호한 부사와 형용사 (거의, 항상, 눈에띄게, 최소의, 가능 하다면, 적절하게 등)
  • 부정적 표현 (솔루션이 제공하지 못하는 것에 대한 구문)
