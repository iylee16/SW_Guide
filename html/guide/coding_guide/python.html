소개

본 문서는 Python 프로그래밍 언어의 소스 코드에 대한 POSCOICT의 코딩 가이드를 정의한 것입니다.

프로젝트가 점점 복잡해지고 규모가 커지면서 개발자가 혼자 코드를 작성하기 보다는 여러 개발자들이 함께 작업하면서 서로 협업하여 시스템을 만들어 나가는 것이 일반화되었습니다 이러한 상호간의 공동작업이 중요해지면서 자신의 코드뿐만 아니라 타 개발자의 코드도 함께 분석하고 리뷰하여 문제점을 함께 파악하고 공동으로 해결해 나가는 경우가 많아지고 있는 추세입니다.

이러한 개발환경에서 코드 스타일을 통일하는 것은 소스코드의 가독성을 높이고 오류의 가능성을 줄이며 개발의 생산성을 높이는데 중요한 역할을 담당하고 있습니다.

본 가이드는 Google 스타일 가이드 기반으로 작성이 되어졌으므로 일반적 Python 프로그래밍의 소스 코드 개발에 문제가 없으며 통상적인 Python Coding 의 표준을 준수한다라고 볼 수가 있습니다.

다만, 포스코ICT 의 특정 개발시 필요한 코딩 가이드 부분은 프로젝트별 요구 사항과 환경을 고려하여 별도 표기로 추가 관리할 예정에 있습니다.

가이드에 기술된 내용은 POSCOICT 각 사업부에서 정보시스템 구현 시 개발자들이 숙독하고 코딩 시 적용할 것을 권고합니다

개발 프로젝트의 경우에 따라 해당 프로젝트에서 요구되어 지는 별도의 코딩 가이드가 있으면 요구되는 코딩 가이드를 우선 적용하며 본 코딩 가이드는 참조 또는 보완용으로 적용하면 됩니다.

Python 언어 규칙

1. 린트

pylint코드를 실행하십시오 .

1.1. 정의

pylint파이썬 소스 코드에서 버그와 스타일 문제를 찾아내는 도구입니다. C 및 C ++과 같이 동적 인 언어가 적 으면 컴파일러에서 일반적으로 발견되는 문제를 찾습니다. 파이썬의 동적 특성 때문에 일부 경고가 잘못되었을 수 있습니다. 그러나 가짜 경고는 매우 드물게 발생합니다.

1.2. 찬성론

오타, 사용 전 바스 (vars-before-assignment) 등과 같은 쉽게 누락 된 오류를 잡아냅니다.

단점

pylint완벽하지 않습니다. 그것을 활용하기 위해 우리는 때때로 다음을해야 할 것입니다 : a) 그 주위를 써라 b) 경고를 억제하거나 c) 그것을 향상시킨다.

1.3 결정

pylint코드 를 실행 해야합니다.

다른 문제가 숨겨지지 않도록 부적절한 경우 경고를 표시하지 않습니다. 경고를 표시하지 않으려면 라인 수준 주석을 설정할 수 있습니다.

dict = 'something awful'  # Bad Idea... pylint: disable=redefined-builtin

pylint경고는 상징적 인 이름 ( empty-docstring)으로 식별됩니다. Google 고유의 경고는로 시작합니다 g-.

상징적 인 이름에서 억압의 이유가 명확하지 않은 경우 설명을 추가하십시오.

이런 방식으로 억압하면 억압을 쉽게 검색하고 다시 방문 할 수 있다는 이점이 있습니다.

다음 pylint을 수행 하여 경고 목록을 가져올 수 있습니다 .

pylint --list-msgs

특정 메시지에 대한 자세한 정보를 얻으려면 다음을 사용하십시오.

pylint --help-msg=C6409

pylint: disable비추천 된 이전 양식을 선호 합니다 pylint: disable-msg.

사용되지 않은 인수 경고는 함수 시작 부분에있는 변수를 삭제하여 생략 할 수 있습니다. 항상 삭제 이유를 설명하는 설명을 포함하십시오. "Unused."로 충분합니다. 예 :

def viking_cafe_order(spam, beans, eggs=None):
    del beans, eggs  # Unused by vikings.
    return spam + spam + spam

이 경고를 표시 _하지 않는 다른 일반적인 형태로는 ' '을 사용하지 않는 인수의 식별자로 사용하고 인수 이름 앞에 ' unused_'를 붙이 거나 ' ' '에 할당하는 방법이 있습니다 _. 이러한 양식은 허용되지만 더 이상 권장되지 않습니다. 처음 두 개는 이름으로 인수를 전달하는 호출자를 중단하지만 마지막 인수는 인수가 실제로 사용되지 않는다고 강요하지 않습니다.

2. Imports

import개별 클래스 또는 함수가 아닌 패키지 및 모듈에만 s를 사용하십시오 . 입력 모듈 에서 가져 오기에 대한 명시적인 면제가 있음에 유의하십시오.

1.1. 정의

한 모듈에서 다른 모듈로 코드를 공유하는 재사용 메커니즘

1.2. 찬성

네임 스페이스 관리 규칙은 간단합니다. 각 식별자의 출처는 일관된 방식으로 표시됩니다. x.Obj객체 Obj가 모듈에 정의되어 있다고 말합니다 x.

1.3. 단점

모듈 이름은 여전히 ​​충돌 할 수 있습니다. 일부 모듈 이름은 불편할 정도로 길다.

1.4 결정

  • import x패키지 및 모듈 가져 오기에 사용 합니다.
  • from x import y여기서는 x접두어 y가없고 모듈 이름 앞에 접두사가없는 패키지 접두어를 사용하십시오 .
  • from x import y as z모듈 두 개 y를 가져 오거나 y불량한 긴 이름 인 경우 사용하십시오 .
  • 표준 약어가있는 import y as z경우에만 사용하십시오 z(예 : npfor numpy).

예를 들어 모듈 sound.effects.echo을 다음과 같이 가져올 수 있습니다.

pylint --help-msg=C6409
from sound.effects import echo
...
echo.EchoFilter(input, output, delay=0.7, atten=4)

가져 오기에서 상대 이름을 사용하지 마십시오. 모듈이 동일한 패키지에 있어도 전체 패키지 이름을 사용하십시오. 이렇게하면 의도하지 않게 패키지를 두 번 가져 오는 것을 방지 할 수 있습니다.

로부터 Imports 입력 모듈은 이 규칙에서 제외됩니다.

3 패키지

모듈의 전체 경로 이름 위치를 사용하여 각 모듈을 가져옵니다.

2.3.1 찬성

모듈 검색 경로가 작성자가 예상하지 않았던 모듈 이름 또는 잘못된 가져 오기의 충돌을 피하십시오. 모듈을 더 쉽게 찾을 수 있습니다.

단점

패키지 계층 구조를 복제해야하기 때문에 코드 배포가 어려워집니다. 현대 배포 메커니즘의 문제는 아닙니다.

2.3.3 결정

모든 새 코드는 전체 패키지 이름으로 각 모듈을 가져와야합니다.

Imports는 다음과 같아야합니다 :

예:

# Reference absl.flags in code with the complete name (verbose).
import absl.flags
from doctor.who import jodie

FLAGS = absl.flags.FLAGS
# Reference flags in code with just the module name (common).
from absl import flags
from doctor.who import jodie

FLAGS = flags.FLAGS

아니요 (이 파일은 현재있는 doctor/who/위치에 있다고 가정 jodie.py)

# Reference flags in code with just the module name (common).
from absl import flags
from doctor.who import jodie

FLAGS = flags.FLAGS
# Unclear what module the author wanted and what will be imported.  The actual
# import behavior depends on external factors controlling sys.path.
# Which possible jodie module did the author intend to import?
                            import jodie

기본 바이너리가 위치한 디렉토리는 sys.path일부 환경에서 일어나는 일에도 불구하고 들어 있는 것으로 간주되어서는 안됩니다 . 이 경우 코드는 로컬이 아닌 import jodie타사 또는 최상위 패키지 를 가리키는 것으로 가정해야합니다 .jodiejodie.py

2.4 예외

예외는 허용되지만 신중하게 사용해야합니다.

2.4.1 정의

예외는 오류 또는 기타 예외적 인 조건을 처리하기 위해 코드 블록 제어의 정상적인 흐름을 벗어나는 수단입니다.

2.4.2 찬성

정상 동작 코드의 제어 흐름은 오류 처리 코드에 의해 복잡하지 않습니다. 또한 특정 조건이 발생할 때 제어 흐름이 여러 프레임을 건너 뛸 수 있습니다 (예 : carry-through 오류 코드 대신 한 단계로 N 개의 중첩 된 함수에서 복귀).

2.4.3 단점

제어 흐름이 혼란을 야기 할 수 있습니다. 라이브러리 호출을 할 때 오류 사례를 쉽게 찾을 수 있습니다.

2.4.4 결정

예외는 특정 조건을 따라야합니다.

3.11 파일과 소켓

파일과 소켓을 완료하면 명시 적으로 파일과 소켓을 닫습니다.

파일, 소켓 또는 기타 파일과 유사한 객체를 불필요하게 열어두면 다음과 같은 많은 단점이 있습니다.

  • 파일 기술자와 같은 제한된 시스템 리소스를 사용할 수 있습니다. 이러한 객체를 많이 다루는 코드는 사용 후 시스템에 즉시 반환되지 않으면 불필요하게 소모 될 수 있습니다.
  • 파일을 열어두면 이동 또는 삭제와 같은 다른 작업이 수행되지 않을 수 있습니다.
  • 논리적으로 닫힌 후에 프로그램 전체에서 공유되는 파일과 소켓이 실수로 읽히거나 쓰여질 수 있습니다. 실제로 닫히면 읽거나 쓰려는 시도가 예외를 throw하여 문제를 빨리 알 수 있습니다.

또한 파일 객체가 소멸 될 때 파일과 소켓이 자동으로 닫히지 만 파일 객체의 수명을 파일 상태와 연결하는 것은 다음과 같은 이유로 실용적이지 못합니다.

  • 런타임이 실제로 파일의 소멸자를 실행할시기는 보장 할 수 없습니다. 다른 파이썬 구현은 지연된 가비지 콜렉션과 같은 다른 메모리 관리 기술을 사용합니다.이 기술은 임의로 그리고 무한정으로 객체의 수명을 늘릴 수 있습니다.
  • 파일에 대한 예기치 않은 참조로 인해 의도 한 것보다 오래 걸릴 수 있습니다 (예 : 예외 추적, 전역 내부 등).

파일을 관리하는 가장 좋은 방법은 "with"문을 사용하는 것입니다. 

with open("hello.txt") as hello_file:
    for line in hello_file:
        print(line)

"with"문을 지원하지 않는 파일과 같은 객체의 경우 contextlib.closing()다음을 사용하십시오 .

import contextlib

with contextlib.closing(urllib.urlopen("http://www.python.org/")) as front_page:
    for line in front_page:
        print(line)

    • 한편으로는 안전과 선명도 사이의 균형을 잘 맞추기 위해 판단을 사용하고, 다른 한편으로는 유연성을 얻으십시오.
    • 유형 관련 오류 (이전의 버그 또는 복잡성)가 발생하기 쉬운 코드에 주석을 답니다.
    • 이해하기 어려운 코드에 주석을 답니다.
    • 유형 관점에서 볼 때 코드가 안정화 될 때 코드에 주석을 답니다. 많은 경우 유연성을 잃지 않으면 서 성숙한 코드의 모든 기능에 주석을 달 수 있습니다.
