테스트 설계 활동 ETVX

ETVX(Entry, Task, Verify, eXit)로 표현되며 착수기준(Entry), 작업절차(Task), 승인(Verify)

테스트 계획 활동 개요 테스트 설계 활동은 개발 산출물을 이용하여 테스트 케이스를 식별하고 정의하는 활동이다.
각 테스트 레벨 및 테스트 유형별로 테스트 케이스를 식별하고 정의하는 시점이 다르며 테스트 베이시스(Test Basis)로 사용하는 산출물이 다르다. 테스트를 성공적으로 수행하기 위해서는 의미 있는 테스트 케이스를 식별하고 정의하는 것이 매우 중요하다.

※ 상세 설계 활동은 별도의 각 테스트 단계별 "테스트 가이드라인" 참고

※ 운영 테스트를 위한 Test Case는 일반적으로 이전 테스트 단계에서 도출된 동일 Case를 활용 또는 유사/부분적인 Test Case를 활용한다.

  1. 테스트 케이스 식별

    테스트 설계자는 프로젝트 특성, 테스트 전략 및 해당 테스트 계획서를 기반으로 테스트 케이스를 식별하고 작성한다.

    단, 단위 테스트 케이스 설계는 일반적으로 개발자가 수행한다.

    ※ 테스트 케이스 도출에 대한 상세한 방법은 별도의 "테스트 케이스 도출 가이드라인" 참고

    단위 테스트
    • 테스트 전략과 단위 테스트 계획서를 통해 테스트 범위를 이해
    • 프로그램 명세서를 통해 상세 기능을 이해
    • 시스템 개발 시 준수해야 하는 표준을 이해
    • 개발자는 담당 모듈에 대한 단위 테스트 케이스를 식별하고 작성
    • 테스트 설계자는 식별된 테스트 케이스를 검토
    통합 테스트
    • 테스트 전략과 결합 테스트 계획서를 통해 결합 테스트 범위를 이해
    • 요구사항명세서, 업무 설계 문서를 통하여 시스템관점에서 테스트 대상 기능을 이해
    • 어플리케이션 설계서/화면설계서/시스템 Interface설계서/DB 설계서 등과 같은 설계 문서를 통하여 통합관점에서의 테스트 대상 기능을 이해
    • 시스템 설계 표준을 이해
    • 통합 테스트케이스를 식별하고 작성
    • 프로젝트 관리자는 작성된 통합 테스트 케이스를 검토
    시스템 테스트
    • 테스트 전략과 시스템 테스트 계획서를 통해 시스템 테스트 범위를 이해
    • 요구사항 명세서, 기능 설계서 등을 통해 시스템 기능을 이해
    • 시스템 설계 표준을 이해
    • 시스템 테스트케이스 작성 : 기능, 비기능  
    • 프로젝트 관리자는 시스템 테스트 케이스를 검토
  2. 테스트 스텝 및 검증 포인트(Verification Point) 정의

    테스트 설계자는 식별된 테스트 케이스를 기반으로 테스트 케이스 실행 순서인 테스트 스텝과 테스트 케이스 성공(PASS)/실패(FAIL)를 판가름할 수 있는 스텝을 지정하여 검증 포인트 정의한다.  

    단, 단위 테스트의 경우에는 일반적으로 개발자가 수행한다.

    필요에 따라, 업무 지식과 테스트 능력을 가진 전문 테스트 엔지니어가 정의할 수 있다.

    ※ 테스트 케이스 도출에 대한 상세한 방법은 별도의 "테스트 케이스 도출 가이드라인"  참고

  3. 테스트 케이스 속성 정의

    테스트 케이스 별로 작성자, 작성일, 테스트 케이스 ID, 설명, 기대결과 등의 속성을 템플릿에 게 정의한다.

    단, 단위 테스트의 경우에는 일반적으로 개발자가 수행한다.

    필요에 따라, 업무 지식과 테스트 능력을 가진 전문 테스트 엔지니어가 정의할 수 있다.

    • 테스트 케이스 명세서를 확인한다.
    • 양식에 맞게 테스트 케이스 속성을 작성한다.
    • 작성된 테스트 케이스 명세서를 검토한다.
  4. 테스트 데이터(Test Data) 준비 및 검증

    테스트 데이터 준비 및 검증은 상위 레벨의 테스트 케이스들을 실행 가능한 형태의 실 데이터로 바꾸는 작업(즉, 테스트 케이스에 따른 실제 입력 데이터를 준비)이다.

    생성된 테스트 데이터 역시 문서화 하거나 또는 경우에 따라 테스트 케이스까지만 문서화할 수 있다.

    필요에 따라, 업무 지식과 테스트 능력을 가진 전문 테스트 엔지니어가 수행할 수 있다.  단위 테스트의 경우에는 일반적으로 개발자가 수행한다.

    • 테스트 케이스를 실행하기 위한 테스트 데이터를 준비한다.
    • 필요 시 테스트 데이터를 커스터마이징 한다.
    • 테스트 설계자는 특성에 맞게 추가/수정한 테스트 데이터를 검증한다.
  5. 테스트 케이스 명세서 작성

    테스트의 효율적이고 효과적인 실행을 위하여 연관성 있는 테스트 케이스를 순서에 맞게 구성하고 테스트 케이스 명세서에 대한 속성을 정의한다.

    • 모듈(영역별) 단위로 테스트 케이스 분류한다.
    • 테스트 케이스 실행 순서를 결정한다.

    테스트를 실행하는 엔지니어가 다른 경우, 작성된 테스트 케이스 명세서를 확인하여 실행 가능한 수준인지를 판단한다.

  6. 테스트 케이스 명세서 검토

    테스트 설계자는 프로젝트 관리자 및 품질보증 담당자등과 검토 및 협의를 수행한다.

    검토 의견이 있을 경우, 테스트 설계자는 합의하에 검토 의견을 반영한다.

  7. 테스트 케이스 명세서 승인

    프로젝트 관리자는 작성된 테스트 케이스에 대한 승인을 프로젝트 오너로부터 획득한다.
