표준 SW 테스트 프로세스는 개발공정에서 효율적이고 효과적인 테스트 수행을 위하여

  • 테스트전략을 수립하고,
  • 테스트를 계획하고
  • 테스트 케이스를 설계하며
  • 테스트 실행을 준비하고
  • 테스트 진행 상태를 확인하고 평가하는 

활동과 세부 절차들로 구성된다.

테스트 전략 활동 ETVX

ETVX(Entry, Task, Verify, eXit)로 표현되며 착수기준(Entry), 작업절차(Task), 승인(Verify)

테스트 전략 활동 개요 테스트 전략 활동은 일반적으로 SW개발 프로세스의 요구사항 분석단계 말에서 이루어지며 사용자 요구사항이 파악되는 시점에서 개발시스템의 Risk, 범위, 담당자 지정, 확보된 예산 등을 고려하여 향후 수행되어질 필요가 있는 테스트에 대한 전체 청사진(Blueprint)을 구상하는 단계라고 볼 수 있으며 이러한 활동의 결과로 마스터 테스트 계획서를 작성한다. 
프로젝트 및 시스템 특성을 고려하여 적합한 테스트 목표 및 실행 전략, 테스트 범위(scope), 접근 방법(approach), 테스트 수행 시 요구되는 자원 (툴 포함), 테스트 활동 및 절차, 테스트 수행 조직(Staffing), 테스트 일정(Schedule), 구성원의 교육요건 등을 사전에 정의하여 프로젝트 수행 시 원활한 테스트 수행을 위한 사전 활동이다.
TAKS
  1. 테스트 요소 식별
  2. 테스트 범위 정의
  3. 테스트 단계별 접근방법(단계 및 수행방안) 정의
  4. 품질 목표 수준 정의
  5. 테스트 추진 체제 정의
  6. 테스트 산출물 및 책임자 정의
  7. 테스트 일정 계획 수립
  8. 마스터 테스트 계획서 작성
  9. 마스터 테스트 계획서 검토
  10. 마스터 테스트 계획서 승인

  • 프로젝트 계획서 작성
  • 테스트(방법론, 필요시 자동화 도구) 관련 교육을 실시 완료함
 입력물

  • 요구사항추적표

  • 테스트 전략 가이드라인
  • 마스터 테스트 계획서
종료조건
  • 검토되고 승인된 테스트 전략
 출력물
  • 갱신된 마스터 테스트 계획서
비고 테스트 전략 활동 적용 기준은 표준 개발방법론의 테일러링 가이드를 따른다

테스트 설계자(테스트 관리자)는

  • 테스트 요소
  • 테스트 범위
  • 테스트 단계 및 수행방법(접근방법)
  • 테스트 추진 체제
  • 산출물 및 책임자 

등의 내용을 "마스터 테스트 계획서" 템플릿을 이용하여 작성하고 프로젝트 관리자(PM)의 검토 및 프로젝트 오너(Owner)의 승인을 받는다.

※ 상세 테스트 전략 활동은 별도 "테스트 전략 가이드라인"참조

  1. 테스트 요소 식별

    테스트 설계자는 개발 시스템 관련 위험을 평가하고 그 위험과 관련된 테스트 요소를 정의한다.

    테스트 수행 시 제거되어야 하는 시스템 관련 위험이나 이슈를 테스트 요소라 하며 테스트 요소 자체가 아니지만 비즈니스 위험으로 변환될 수 있다. 테스트 요소를 식별하기 위해 다음과 같은 활동이 필요하다.

    테스트 요소 식별
    • 테스트 대상 시스템 및 관련 비즈니스 위험을 식별
    • 테스트 수행 중에 평가되어야 하는 위험을 기반으로 테스트 요소를 식별
    • 심각도 및 위험도를 고려하여 테스트 요소를 분류
    식별된 테스트 요소에 대한 테스트 단계/유형별 테스트 전략 정의
    • 개발 프로젝트의 생명주기(Lifecycle)을 파악
    • 테스트 요소와 관련 위험 최소화를 위해 개발프로젝트의 생명주기 각 단계에 적합한 테스트 유형을 정의
    • 테스트 유형별 테스트 방법을 정의
  2. 테스트 범위 정의

    테스트 설계자(관리자)는 개발프로젝트의 범위 및 상위 요구사항을 분석하여 테스트 범위를 설정한다.

    테스트를 통하여 적합성을 검증 받아야 할 테스트 범위(Scope)는 대상 시스템 전체가 될 수도 있고, 경우에 따라 일부분이 될 수도 있다.

    선정된 테스트 범위는 향후 단계별 테스트에 대한 상세 계획 수립 시, 요구분석 및 설계 단계에서 정의된 각 테스트 요구사항 들을 검토하여 테스트 항목, 테스트할 항목 특성과 테스트하지 않을 항목 특성을 명세화 하는 기반 자료로 활용된다.

  3. 테스트 단계별 접근방법 정의

    테스트 설계자(관리자)는 프로젝트에서 적용할 테스트 단계와 단계별 테스트에 대한 수행방안을 정의한다.

    개발생명주기별 테스트 단계 정의
    • 표준적인 테스트 단계(단위/통합/시스템/운영 테스트)를 기반으로 개발 프로젝트에서 수행할 테스트 단계 정의(표준 테스트 단계 결합, 생략 또는 세분화)

    ※ 테스트 할 시스템 크기와 복잡도, 시스템 중요도, 개발 성숙도, 고객 요청사항, 테스트 일정, 테스트 환경, 테스트 참여 인원 및 조직과 경험, 위험 등을 고려

    각 단계별 테스트 수행방안 정의
    • 프로젝트에서 수행될 각 테스트 단계별로 필요한 테스트 유형 및 어떻게 테스트를 수행할 것인지에 대한 방안을 정의

    필요에 따라 아래와 같은 요소들을 고려할 수 있다.

    • 블랙박스 또는 화이트박스 테스트 기법
    • 상향식(Bottom-Up) 및 하향식(Top-Down), 백본(Back-Bone), 빅뱅(Bing-Bang) 등의 테스트 접근법
    • 테스트 자동화 도구 또는 전문 도구에 의한 테스트 공정관리가 요구 되는 경우, 프로젝트의 요구사항을 파악하여 전문 테스트 도구를 선정하고, 반복적이고 리스크가 높은 부분을 중점적으로 자동화 테스트 범위로 선정
  4. 품질 목표 수준 정의

    테스트 단계에서 수집∙관리가 필요한 품질 지표(예: 결함율, 결함밀도, 자동화율, 테스트 케이스 적절성, 테스트 정확성, 응답시간, 서버 처리율 등) 및 목표 수준을 정의한다.

    품질 목표 수준 정의는 고객요청 또는 조직에서 정의한 목표를 기준으로 개발계획서 ‘2. 프로젝트 성과 목표’에 기술하는 것으로 대체할 수 있다.

    프로젝트 관리자 또는 테스트 설계자는 테스트 활동 시 해당 데이터를 수집하여 측정지표로 관리하여야 한다. 또한 필요시, 테스트 단계 및 유형별로 품질 지표 및 목표 수준을 상세히 정의할 수 있다.

  5. 테스트 추진 체제 정의

    상위 단계의 TASK에서 정의한 테스트 단계 및 접근방안에 따라 테스트를 수행할 추진 조직 및 역할을 정의한다.

  6. 테스트 산출물 책임자 정의

    테스트 종류별 산출물을 선정하고, 작성 시점 및 책임자를 정의한다.

  7. 테스트 일정 계획 수립

    각 테스트 단계에 대한 전체적인 일정을 기술한다. 상세 일정 수립이 어려운 경우는 각 테스트 전체 일정을 기술하고 상세 일정은 단계별 테스트 계획 활동에서 수립, 해당 계획서에 기술한다.

  8. 마스터 테스트 계획서 작성

    테스트 설계자(관리자)는 상위 단계의 TASK를 기반으로 하여, 마스트 테스트 계획서를 작성한다.

    마스터 테스트 계획서에는 식별한 테스트 범위, 테스트 단계 및 수행방안, 테스트 추진 체제, 테스트 산출물 및 책임자, 테스트 일정과 같은 테스트 전략에 대한 내용을 "마스터 테스트 계획서" 템플릿에 기술한다.

  9. 마스터 테스트 계획서 검토(테스트 전략 협의)

    테스트 설계자는 마스터 테스트 계획서에 기술된 테스트 전략에 대하여 프로젝트 관리자 및 품질보증 담당자와 검토 및 협의를 수행한다.

    검토 의견이 있을 경우, 테스트 설계자는 합의하에 검토 의견을 마스터 테스트 계획서에 반영한다.

  10. 마스터 테스트 계획서 승인

    프로젝트 오너는 마스터 테스트 계획서를 승인한다.

    프로젝트 오너는 마스터 테스트 계획서대로 테스트가 수행되면 프로젝트 및 프러덕트의 품질을 보증할 수 있는지 여부를 판단하고 작성된 마스터 테스트 계획에 대하여 승인한다.

  11. 테스트 일정 계획 수립

    각 테스트 단계에 대한 전체적인 일정을 기술한다. 상세 일정 수립이 어려운 경우는 각 테스트 전체 일정을 기술하고 상세 일정은 단계별 테스트 계획 활동에서 수립, 해당 계획서에 기술한다.

테스트 계획 활동 ETVX

ETVX(Entry, Task, Verify, eXit)로 표현되며 착수기준(Entry), 작업절차(Task), 승인(Verify)

테스트 계획 활동 개요 테스트 계획 활동은 테스트 설계자가 각 단계별 상세 테스트 계획을 수립하는 단계이다.
각 단계별 테스트에 대한 계획은 개발 단계에 따라 작성되는 시점이 다르며 입력되는 산출물도 다르다. 예를 들어, 인수 테스트 계획은 요구사항 분석 단계 완료시점, 시스템 테스트 계획은 Infra 아키텍처 설계 완료시점에 작성 가능하며 결합 테스트 계획은 설계 단계 완료시점에 작성 가능하다. 단위 테스트계획서는 제작 단계에 작성 가능하다.

※ 테스트 전략 활동을 수행하는(마스터 테스트 계획서 작성) 프로젝트는 마스터 테스트 계획서를 기반으로 하여 상세 테스트 계획을 수립한다.

단위 테스트/통합/시스템/운영 테스트 계획 절차는 테스트 단계와 상관없이 동일하다.

※ 각 단계별 테스트 계획 활동에 대한 상세 내용은 별도 "테스트 수행 가이드라인" 참조

  1. 테스트 범위 정의

    테스트 전략을 기반으로 테스트 범위를 정의한다. 모든 대상 영역을 테스트 하지 않을 경우, 테스트 제외 범위와 제외 사유를 기술한다.

    시스템 테스트의 경우 수행할 비기능 테스트 종류 및 범위를 정의한다.

  2. 테스트 착수 기준과 완료 기준 정의
    테스트 착수 기준을 정의한다.

    테스트 착수 기준은 테스트를 시작하기 전에 완료되어야 할 활동을 말한다.

    결합 테스트의 착수기준의 예로는 결합 테스트 케이스 검토, 승인되고 결합 테스트 환경 및 데이터가 준비 완료되는 것이다.

    테스트 완료 기준을 정의한다.

    테스트 완료 기준은 테스트 활동을 끝마쳐야 할 시기 결정하기 위한 기준을 의미한다.

    테스트 완료 기준의 예로는 계획된 테스트 케이스에 대한 테스트 수행 및 중대한 결함이 모두 조치 완료되는 것이다.

  3. 테스트 환경 정의

    테스트 수행할 환경을 정의한다. 일반적으로 단위 테스트 시에는 별도의 테스트 환경을 구축하지 않는다(개발환경에서 테스트 수행).

    프로젝트 개발계획서와 중복된 내용에 대해서는 해당 참조로 대체할 수 있다.

    • 테스트 환경에 대한 HW 및 SW 구성을 기술한다.
    • 테스트를 수행할 때, 필요한 프로그램 라이브러리, 데이터 라이브러리, 디렉토리 등의 위치를 정의한다.
    • 스트에 필요한 테스트 데이터 생성 및 유지 절차를 정의한다. 일반적으로 단위 테스트의 경우, 필요할 경우에만 기술한다.

    또한 필요하다면 시스템 설치 방법, 데이터베이스 셋업, 필요한 스크립트나 프로그램 설치 절차 및 활동, 접근 권한, 백업 절차를 추가로 정의할 수 있다.

  4. 필요 자원과 일정 정의

    필요 자원은 HW, SW, 인력으로 구분하여 정의하며, 기존 보유 자원도 기술하여야 한다.

    테스트 일정은 테스트 단계의 주요 활동 및 시작/완료일자를 정의한다.

    보통 프로젝트 수행 일정을 따르며, 테스트 범위가 변경될 경우 테스트 일정이 변경될 수 있다.

    각 주요 활동간의 의존성과 해당 활동 수행에

    • 필요한 HW와 SW 자원을 정의한다.
    • 테스트 수행에 필요한 인력을 정의한다.
    • 테스트 활동에 대한 상세 일정을 수립한다.
      • 수행활동간의 의존성과 요구되는 제약사항들을 고려하여 테스트를 준비하고 이행하는데 필요한 작업들을 식별하여 기술한다.
        (예 : 테스트 대상 그룹의 가용성 여부, 필요 자원 준비, 테스트 환경 준비, 기타 제약사항 등을 테스트 활동에 대한 상세 일정을 명세)
      • 작업을 담당할 책임자, 시간 완료일 등을 수립
      • 필요시, 테스트 데이터(DB)의 생성을 위한 데이터 전환 일정을 고려
  5. 테스트 고려사항 정의

    개발프로젝트 특성상 필요할 경우 테스트 수행에 있어서의 고려사항 및 위험요소에 대한 사항을 정의한다.

  6. 결함 및 이슈 보고 절차 정의

    테스트과정에서 발견된 결함 및 이슈 보고 절차를 정의한다.

    보고 절차에는 발견된 결함 기록, 관련자의 정보 공유, 해결하기 위한 절차가 포함된다.

    일반적으로 해당 개발 프로젝트의 프로젝트 위험 및 이슈관리 지침을 따른다.

    별도의 결함 및 이슈 보고절차가 필요할 프로젝트는 상세히 기술한다.

  7. 테스트 계획서 작성

    테스트 설계자는 테스트 단계별로 상위 단계의 Task를 기반으로 하여, 상세한 테스트 계획서를 해당 "테스트 계획서" 템플릿에 기술한다.

    단, 개발프로젝트 외부 조직과 협업이 필요한 경우에는 긴밀하게 협의 후 작성한다.

  8. 테스트 계획서 검토

    테스트 설계자는 프로젝트 관리자(PM) 및 품질보증담당자(QA)와 검토 및 협의를 수행한다.

    검토 의견이 있을 경우, 테스트 설계자는 합의하에 검토 의견을 반영한다.

  9. 테스트 계획서 승인

    프로젝트 관리자는 작성된 테스트 계획서에 대한 승인을 프로젝트 오너로부터 획득한다.

    프로젝트 오너는 테스트 계획서가 프로젝트 및 프로덕트(제품,시스템)의 품질을 보증할 수 있도록 작성되었는지 여부를 판단하고 작성된 테스트 계획에 대하여 승인한다
