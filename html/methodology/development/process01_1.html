활동(Activity)명 2.1 비즈니스 로직 분석
목적

기술이 배제된 상태에서 업무를 이해하고 분석하고 명세하는 활동이며, 이 활동은 추상활동 또는 그룹활동으로 구체적인 세 가지 활동을 대표하는 활동임

  • 시스템 분석가들이 도메인에 포함된 업무를 이해하고 이를 시스템 관점에서 분석함
  • 설계로 들어가기 전에 업무의 범위와 내용을 정의함
절차
  1. 도메인 객체 모델링
  2. 컴포넌트 명세
  3. POC 프로토타입
역할자 시스템 분석가, 시스템 설계자
산출물 분석 모델(분석 도메인 모델 + 컴포넌트 명세 모델), POC 프로토타입 결과
가이드
  • 도메인 객체 모델링
    • 절차
      1. 핵심 개념은 무엇인지?
      2. 어떤 명사들이 존재하는지?
      3. 개념들 간의 관계는 어떻게 되는지?
    • 가이드
      • 핵심 개념과 보조 개념을 식별한다.
      • 동적인 측면의 서비스와 정적인 도메인 객체를 명확히 분리한다.
      • 업무의 정적인 즉 개념구조 부분은 도메인 객체 모델로 표현하고, 동적인 부분은 컴포넌트 모델(=컴포넌트 명세)를 이용하여 표현한다. 이 두 부분을 명확하게 분리하지 않으면
      • 시스템 실행 관점이 아닌 업무 관점에서 문제를 바로 보고 이해한다. 이 단계에서 기술은 존재하지 않으며 어떤 기술을 사용하여 시스템을 구축할 지 알 수 없다. 바로 옆에서 아키텍처 팀이 열심히 그러한 고민을 하고 있음을 기억한다.
      • 도메인 객체 모델링에는 설계자 및 핵심 개발자가 참여하여 함께 모델링을 하여야 한다. 그러지 않을 경우, 분석자가 설계자에게 또 설계자가 개발자에게 이해한 개념을 넘기는 사이에 개념의 왜곡이 충분히 일어날 수 있기 때문이다.
      • 애매한 개념은 반드시 POC 프로토타입을 통해서 검증을 하여야 한다.

        예 ) 낙찰가라는 개념이 무엇이며 어떻게 결정되는가?

  • 컴포넌트 명세
    • 절차
      1. 유스케이스 명세서 상세화
      2. 유스케이스 이벤트 흐름에서 시스템의 리액션을 서비스로 식별
      3. 이렇게 식별한 서비스를 유형별로 묶어서 컴포넌트 식별
      4. 컴포넌트 간의 관계를 정의하고 이 과정에서 인터페이스 상세화 함
  • POC 프로토타입 개발
    • 절차
      1. 도메인 객체 모델 정제 및 상세화
      2. 프로그래밍
활동(Activity)명 2.2 UI 분석
목적

시스템의 사용자를 조사하여 분석하고 개략적인 화면 간의 이동을 분석하여 주요 UC 스토리보드와 프로토타입을 개발하는 활동임

  • 시스템을 사용하는 사용자와 시스템의 개략적인 네비게이션의 흐름을 이해하고 분석함

설계로 들어가기 전에 주요 UC 스토리보드를 정의하고 UI 프로토타입을 개발함
절차
  1. 사용자 분석
  2. 네비게이션 분석
  3. 주요 UC 스토리보드 작성
  4. 주요 UC 프로토타입 개발
역할자 UI Designer
산출물 페르소나, 스토리보드, 프로토타입
가이드
  • 사용자 분석
    • 절차
      1. 사용자 조사(User research)
      2. 사용자 세분화(User segmentation)
      3. 페르소나(pefsona) 작성
    • 가이드
      • 사용자 조사는 인터뷰 또는 설문조사방법 등으로 사용자의 요구(needs), 기대, 경험, 행동패턴, 정황 등의 데이터를 수집하고 정리한다.
      • 정리된 사용자 조사 자료를 근거로 비즈니스 목표, 시스템의 목적, 사용자 유형 등을 고려해 사용자 세분화(user segmentation)을 진행하고 타겟 유저(target user)를 개발한다.
      • 세분화된 사용자 그룹을 통해 다양한 사용자 유형을 대표하는 가상의 캐릭터인 페르소나(persona)를 개발하고, 페르소나에 사용자에 대한 다차원적인 모습을 묘사함으로 기획 및 디자인의 유효성에 대한 판단의 근거 자료를 제공해야 한다.
      • 개발된 페르소나는 사용자에 대한 프로젝트 이해관계자들의 상이한 관점의 갭을 줄인다.
      • 페르소나의 목적은 사용자의 요구, 목표를 대표하는 사람들의 행동을 설명하는 것이지 사용자 집단의 경향을 설명하는 것은 아니다.
      • 또한, 사용자 분석을 통해 시스템의 기능, 컨텐츠들의 우선순위를 결정할 수 있음.
  • 네비게이션 분석
    • 절차
      1. 유스케이스와 용어 검토
      2. 기능과 컨텐츠에 대한 상, 하위 수준 검토
      3. 메뉴 구조 결정, 메뉴 또는 기능간 이동 경로 결정
    • 가이드
      • 시스템의 기능과 컨텐츠, 시스템의 목표, 사용자의 요구에 대한 이해가 전재된 이후 진행해야 한다.
      • 전체적인 시스템의 구조와 이동경로를 시각화하여 표현하는 것이 프로젝트 이해관계자의 이해를 돕는데 바람직하다.
      • 단순한 페이지의 이동을 표현하는 것이 아니라 페이지 간의 의미관계, 기능, 사용자의 심성모형(mental model) 등을 고려하여 작성한다.
      • 개발자 언어가 아닌 사용자의 언어 또는 비즈니스 언어를 사용한다.
      • 필요에 따라 기능별, 의미별로 그룹핑을 지어주고 사용할 용어의 일관된 레이블 기준을 정한다.
      • 상위 레벨의 정보는 사용자의 기대치를 반영하고 하위레벨은 수행할 타스크 유형을 적용한다.
      • 대부분 초기에 확정된 사이트 맵은 향후 설계, 구현단계에 많은 영향을 주므로 반복적인 개발이 필요하고 고객의 승인을 거치는 과정이 필요하다.
  • UC 스토리 보드 작성
    • 절차
      1. 유스케이스 분석, 사이트 맵 검토
      2. 스토리보드 작업도구 선정
      3. 주요 UC 스토리보드 작성(low fidelity)
    • 가이드
      • 상세한 스토리보드를 작성하는 것보다 주요 유스케이스의 기능 위주로 개략적인 화면을 작성하는 것이 바람직하다. 개발된 스토리보드는 프로토타이핑 기법을 통해 재검증 될 것이기 때문에 상위수준에서 하위수준으로 반복적이고 점진적으로 개발된다.
      • 네비게이션 분석에서 개발된 사이트 맵을 참조하여 이동 경로를 고려한다.
      • 사용자 분석을 통해 얻은 페르소나를 주인공으로 역할을 부여하여 개발자 입장이 아닌 사용자 입장에서 스토리보드를 작성해야 한다.
      • 결정되지 않은 항목에 대해서는 고민하지 말고 컨텐츠 또는 기능이 배치될 영역만 확보한다.
      • 초기 스토리보드는 디지털 방식으로 개발될 필요는 없으며 종이 또는 화이트보드를 이용해서 프로젝트 이해관계자들과 협의를 통해 개선한다.

      스토리보드 작성도구의 선정은 작업자의 도구에 대한 이해도와 숙련도, 시스템 범위 등을 고려한다

  • UC 프로토타입 개발
    • 절차
      1. 주요 UC 스토리보드 검토
      2. 초기 프로토타이핑 도구 선정
      3. 프로토타이핑 수행 및 평가
    • 가이드
      • 프로토타입 도구는 가급적 손쉽게 빨리 프로토타입을 진행할 수 있는 도구를 선정함.
      • 스토리보드를 이용하여 프로토타입을 할 수 있으나 시뮬레이션이 가능하다면 보다 효과적임.

    프로토타입을 어느 충실도(fidelity)까지 할 것인지를 사전에 결정하고 기준에 맞춰 진행해야 함
활동(Activity)명 2.3 테스트 계획서 작성
목적

분석 단계를 통해 식별된 기능 및 비기능 요구사항에 대해 SW 시스템 설계 명세가 요구사항에 만족하는지 검증하고 보장하기 위해 테스트 전략을 포함하여 계획서를 작성함

  • 시스템을 테스팅하기 위한 체계적인 접근 방식을 계획함.

테스트할 대상을 사전에 식별하여 그에 맞는 테스트 전략을 수립함
절차
  1. 테스트 영역 식별
  2. 테스트 수행에 대한 전략 마련
  3. 테스트 실행에 필요한 자원 및 책임 식별
  4. 테스트를 통한 위험 및 불확실성 검증 식별
역할자 테스터(리더, 설계자)
산출물 테스트 계획서
가이드

테스트 전략, 테스트 계획, 테스트 베드를 만듦
활동(Activity)명 2.4 테스트 케이스 작성
목적

분석 결과를 토대로 테스트를 수행하는 절차와 방법에 대해 기술함

  • 요구사항을 충족시키는 시스템의 기능 및 비기능에 대해 검증이 가능한 테스트 케이스를 작성함.
  • 각 테스트 케이스 간의 관계를 정의하고 전체 테스트 케이스의 구성을 체계적으로 수립하여 테스트 수행이 원할하게 수행되도록 함.
절차
  1. 테스트 시스템에 대한 아키텍처 정의
  2. 테스트 환경 정의
  3. 구성 요소 간의 관계 정의
  4. 테스트 실행 단계 정의
