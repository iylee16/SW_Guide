활동(Activity)명 5.1 서버 컴포넌트 구현
목적

설계된 서버 컴포넌트를 개발자의 영역에서 이를 올바로 이해하고 시스템이 동작할 수 있도록 프로그래밍 기술을 사용해서 개별 컴포넌트를 구현하는 활동

  • 설계 의도를 올바로 파악하여 컴포넌트 명세에 만족하는 구현체를 프로그래밍 가이드를 준수하여 코드를 작성한다.
  • 개발 환경 하에서 단위테스트 활동과 같이 실행 가능한 산출물(artifact)을 만든다.
  • 구현 활동의 결과로 나타나는 모든 산출물들은 형상관리의 대상이며, 반복 주기에 따라 여러 번 수행되는 활동이다.
절차
  • 도메인 모델 구현
  • 인터페이스 모델 구현
역할자

개발자
산출물
  • 서버 컴포넌트 구현체
가이드
  • 구현 방법과 그에 맞는 테스트 접근 방법을 사전에 결정함.
  • 컴포넌트 설계에 대한 충분한 이해와 설계 의도를 구현 관점에서 파악함.
  • 설계 패턴 및 구현 패턴을 참조하여 가능한 한 확장 가능한 결과물로 구현함.
  • 재사용 가능한 요소들을 최대한 식별하여, 이를 사용할 수 있는 방법을 고려함.
  • 중요한 결정 사항에 대해서 충분히 토의하고, 다양한 제약 조건들에 대해서도 고려하여 구현함.
  • 코드 구현은 가능한 한 빨리 테스트를 수행하고 이에 대한 피드백을 받아 이를 코드에 반영함.
활동(Activity)명 5.2 UI 컴포넌트 구현
목적

UI 설계를 기반으로 사용자 화면과 이를 구현된 서버 컴포넌트와 연결하는 UI 컴포넌트를 구현

  • 사용자 화면의 각 UI 요소별로 화면 표준에 근거하여 이를 구현한다.
  • 네비게이션 설계에 맞는 화면 연결 구조를 구현한다.
  • 화면 기능에 대해서 서버 컴포넌트 구현체와의 연결을 구현한다.
절차
  • 화면 요소 구현
  • 서버 컴포넌트 구현체와의 연결
  • 화면 연결
역할자

개발자
산출물
  • UI 컴포넌트 구현체
가이드
  • 재사용 가능하도록 화면을 구현하고, 기존 화면을 최대한 재사용함.
  • 화면 로직이 너무 복잡해지지 않도록 하며, 서버와의 역할과 책임을 적절하게 분할함.
  • 서버와의 연결 구조는 최대한 느슨하게 연결하는 구조로 구현하고, 테스트시 서버와의 연결없이 화면 기능 테스트가 가능하도록 구현함.
  • 너무 많은 정보를 보여주는 화면을 위해서는 서버와의 통신을 가능한 한 최소화시키고, 적정한 정보의 수준에서 통신하도록 함.
  • UI 컴포넌트 구현시 컴포넌트 인터페이스만을 참조하여 테스트를 수행함.
  • 기존 컴포넌트 명세에 필요한 오퍼레이션이 없는 경우, 기능이 유사한 컴포넌트의 인터페이스를 최대한 사용하며, 이는 컴포넌트의 재사용성을 위해서 UI에 특화된 기능을 최소화시킴.
활동(Activity)명 5.3 코드 리뷰
목적

구현된 코드가 표준과 명세에 준수하여 만들어졌는지를 점검하고, 요구사항을 정의한 내용이 모두 포함되었는지를 점검

  • 명세, 요구사항, 설계 문서에 만족하는 코드가 작성되었는지를 점검함.
  • 코드에 대한 개선 기회를 제안함.
  • 코드와 검토 대상 시스템에 대해 다른 방식의 접근 방법을 습득함.
절차
  • [가능한 경우] 다양한 도구를 통한 코드에 대한 품질 측정
  • 특정 기능의 코드에 대한 의견 교환 및 검토
  • 리뷰 결과에 따른 코드 개선
역할자

개발자
산출물

코드리뷰 결과서
가이드
  • 리뷰를 수행하는 사람은 코드에 대한 잘못을 질책하는 것보다는 가능한 해결책을 제시하도록 함.
  • 설계를 기준으로 올바른 방향으로 구현이 되었는지를 점검하도록 함.
  • 코드를 읽는 이로 하여금 주석(설명)을 보지 않고도 이해가 되도록 코드가 구성되도록 함.
  • 명확하지 않은 코드로 인해 복접성이나 확장성을 침해하는 경우 과감하게 해당 코드에 대한 수정을 가함.
활동(Activity)명 5.4 리팩토링 및 개선
목적

구현 과정에서 파악된 설계 개선 사항에 대해 컴포넌트 명세 및 설계, 코드 등에 대해 개선 작업을 수행

  • 오퍼레이션의 행위는 그대로 두고, 내부 구조의 개선을 통해 확장 및 재사용 관점에서 조정 작업을 수행함.
  • 컴포넌트가 사용되는 영역에 따른 품질 속성에 대해 재구조화를 통해 이를 달성함.
  • 리팩토링을 통한 개선 활동으로 인해 기능 명세 뿐만 아니라, 비 기능 명세에 대해서도 이를 만족하도록 함.
절차
  • 컴포넌트 명세 개선
  • 컴포넌트 리팩토링
역할자

개발자
산출물

(개선된) 컴포넌트
가이드
  • 반드시 단위 테스트 케이스를 작성하고, 리팩토링 후 테스트를 수행해야 함.
  • 리팩토링에 대한 범위는 크게 생각하고, 작은 부분부터 시작함.
  • 리팩토링이 잘못 되었다고 생각되면, 반드시 이전 소스로 복원함.
  • 리팩토링에 대한 내역은 주석으로 반드시 기재함.
  • 도구를 최대한 그리고, 적극적으로 활용함.
  • 임시로 만든 변수나 메소드라 하더라도 의미있고 정확한 용어를 사용함.
  • 너무 긴 소스 코드에 대한 리팩토링 역시 의미가 있지만, 불필요한 코드에 대한 제거도 의미있는 리팩토링임.
  • 해당 컴포넌트가 사용되는 다양한 환경을 이해하여 전체적인 관점에서 리팩토링을 수행함.
  • 리팩토링 후 목적과 목표에 부합되었는지를 반드시 점검함.
활동(Activity)명 5.5 컴포넌트 통합
목적

의존관계에 있는 컴포넌트를 통합하여 기능의 완전성을 달성하며, 통합 과정에서 컴포넌트 간의 의존성을 개선

  • 의존관계를 통해 전체 애플리케이션의 기능을 완성함.
  • 기능 요구사항에 부합하는 컴포넌트 인터페이스의 조정을 통한 의존관계 개선
  • 애플리케이션을 구성하는 컴포넌트 식별을 통한 배치 모델 완성
절차
  • 의존관계에 있는 컴포넌트 식별
  • 단위 애플리케이션 구성을 통한 컴포넌트 통합
  • 시스템 테스트를 통한 검증
역할자

개발자, 설계자
산출물
  • [검증된] 컴포넌트
가이드
  • 컴포넌트 통합을 통한 의존관계의 전체 컴포넌트를 식별함.
  • 테스트 과정에서 필요한 컴포넌트와 실행 과정에서 필요한 컴포넌트를 구분함.
  • 의존관계에 있는 컴포넌트에 대한 버전 충돌에 대한 식별 및 해결함.
  • 의존관계 원칙에 맞게 컴포넌트 의존관계 형성을 점검함.
  • 컴포넌트 통합 후 시스템 테스트를 통해 전체적인 기능이 요구사항에 부합하는지를 검증함.
활동(Activity)명 5.6 단위테스트
목적

컴포넌트 내부 및 외부에 대한 로직 검증을 통해 품질을 향상시키는 작업으로 단위 테스트를 수행하여 코드 커버리지 비율을 일정 수준으로 유지시킴

  • 설계 의도대로 컴포넌트가 구현되었는지를 검증
  • 코드 변경으로 인한 기능의 깨짐 현상을 방지하여 품질을 일정 수준으로 유지함.
  • 반복적이고 자동화된 테스트를 통한 회귀 상태로의 방지.
절차
  • 단위 테스트 대상 식별 및 테스트 접근 방법 결정
  • 단위 테스트 코드 작성 및 실행
  • 소스 코드에 대한 조정[리팩토링] 및 재테스트
역할자

개발자
산출물
  • 단위 테스트 코드
가이드
  • 테스팅은 품질 관리의 일환임.
  • 개발 중인 소프트웨어에 남아 있는 버그가 존재하지 않을 때까지 테스트를 수행함.
  • 단위 테스트는 개발자에게 더 이른 시간에 피드백을 받을 수 있는 메커니즘을 제공함.
  • 작고, 자동화된 테스트를 작성함.
  • 단위 테스트는 소스 코드 작성과, 설계[리팩토링] 작업이 같이 발생되는 활동임.
  • 최대한 디버거를 사용하지 않는 수준의 단위테스트를 작성함.
  • 개발자 자신 작업에 대한 확신감이 들 수 있도록 단위테스트를 작성함.
  • API에 대한 테스트는 최대한 쉽고 간결하게 작성되는 것이 좋음.
  • 코드에 대한 변경은 테스트를 필요로 하며, 이는 단위 테스트를 통한 코드에 대한 건전성을 유지시켜 줌.
  • 테스트할 대상 코드가 너무 크거나 많은 경우에는 설계, 구현 코드의 변경을 통해 테스트가 가능한 형태의 구조나 크기로 변경하고 이를 테스트 수행함.
활동(Activity)명 5.7 데이터 이행
목적

AS-IS 데이터에 대해 신규 시스템에서 변경된 TO-BE 데이터의 형태로 데이터 이관 및 이행을 작업을 수행

  • 기존 데이터를 수집 및 분석을 통해 신규 시스템으로 데이터를 통합 및 이전을 수행함.
  • 데이터 품질 개선을 데이터 이행시 같이 수행함으로써 데이터로 인한 오류를 최대한 방지함.
  • 관리 대상의 데이터를 식별함으로써 이후 운영 단계에서 발생할 수 있는 문제를 사전에 식별함.
절차
  • 이행 환경 구축 / 이행 요구사항 정의
  • 팀 구성 및 업무 할당
  • 이행 설계/개발
  • 이행 환경 구축
  • 이행 통합 테스트
  • 최종 데이터 이행
역할자

개발자, DBA
산출물
  • 데이터 이행 계획서
가이드
  • 사전에 데이터 이행으로 인한 위험을 최대한 식별하고, 그에 맞는 해결 방안을 모색함.
  • 데이터 이행에 대한 검증 및 시험 방안을 마련하고, 이해에 대한 효율성을 점검함.
  • 전반적인 개발 활동과 관련된 데이터 이행에 대해 우선순위를 부여하여 데이터 이행함.
  • 관리 항목 및 관리 코드를 표준화시킴으로써 데이터 이행의 효과성을 최대화시킴.
  • 프로그램 작성과 도구를 사용한 이행에 대한 장단점을 최대한 비교하여 적절한 이행 전략을 마련함.
  • 데이터 이행후 발생될 수 있는 오류 및 위험성을 식별하여 추가 작업을 수행함.
  • 장애 유형별로 대처 방안을 마련하고, 그에 따른 적절한 조치를 수행함.
  • 데이터의 정합성, 일치성, 적시성, 유형성 등을 파악하여검증함.
활동(Activity)명 5.8 테스트 준비
목적

구현된 기능에 대한 테스트를 위한 준비로 테스트 계획에 따른 환경 및 테스트 시스템을 준비하고, 테스트 케이스에 기술된 내용대로 테스트 데이터와 스크립트를 작성

  • 테스트를 수행할 수 있는 환경과 시스템 준비
  • 구현된 기능을 최대한 테스트 수행하기 위해 테스트 케이스의 개선 및 스크립트 작성
