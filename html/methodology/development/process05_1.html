활동(Activity)명 6.1 통합 테스트
목적

전체 컴포넌트가 통합된 상태인 제품화 상태(staged 상태)에서 컴포넌트 간의 상호 관계, 인터페이스 연계 등에 관련된 버그를 점검. 빌드 계획이 같이 수반됨

  • 전체 컴포넌트의 통합으로 기능의 완전성을 점검

컴포넌트간의 의존성 및 인터페이스 연계에 대해 초점을 맞추어서 결함을 찾아냄
절차
  • 테스트 시스템 빌드 및 배포
  • 통합 테스트 수행

활동(Activity)명 6.2 시스템 테스트
목적

전체 시스템에서 발생되는 다양한 유형의 버그를 점검함. 설치 및 사용자 테스팅과 같은 사용자 관점에서의 테스트도 포함됨

  • 전체 시스템에서 발생되는 다양한 유형의 결함을 찾아냄.

기능 이외에 비기능적인 내용까지 모두 포괄하여 테스트를 수행함
절차
  • 테스트 시스템 빌드 및 배포, 설치
  • 시스템 테스트 수행

비기능 테스트

  • 비정상적인 상태에서도 시스템이 적절하게 기능을 수행하는지를 점검
  • 성능/부하 테스팅
    • 성능 테스팅(performance testing)은 특정 부하 상태에서 시스템이나 서브 시스템이 얼마나 빨리 반응하는지를 점검
    • 확장성(scalability), 신뢰성(reliability), 자원 사용에 대한 품질도 측정 가능
    • 부하 테스팅(load testing)은 대량의 데이터나 다수의 사용자와 같은 특정 부하에서 SW가 기능을 제대로 수행하는지를 점검
    • 확장성(scalability)과 관련된 테스팅
      내구성 테스팅(endurance testing) 이라고도 함.
      용량(volume) 테스팅은 기능 측정의 방법. 스트레스(stress) 테스팅은 신뢰성 측정의 방법. 부하(load) 테스팅은 성능 측정의 방법.
  • 안정성(stability) 테스팅
    • 인정할 수 있는 기간 동안 혹은 넘어서 지속적으로 기능을 수행하는지를 점검
    • 부하(load) 테스팅이라고도 함.
  • 사용성(usability) 테스팅
    • 사용자 인터페이스가 사용하기 쉽고 이해하기 쉬운지를 점검
  • 보안성(security) 테스팅
    • 외부 침입자에 의한 시스템 침투를 방어하는 데이터 암호화에 대한 절차 점검
  • 국제화 및 지역화
    • 임의의 지역에 대한 SW 적응성 테스팅
    • 새로운 언어나 문화 환경에서도 시스템이 기능을 제대로 수행하는지를 점검
  • 기타 테스팅
    • 파괴(destructive) 테스팅
      시스템의 견고성을 위해 시스템의 실패를 유발시키는 테스팅
    • 탐험적(exploratory) 테스팅

      테스터의 개인적인 자유와 책임을 강조한 테스팅의 유형으로 프로젝트를 통해 동시에 수행되는 상호 지원 활동으로써 테스트 관련 지식, 테스트 설계, 테스트 실행, 테스트 결과 해석을 다룸으로써 테스트 작업의 품질을 지속적으로 최적화시킴.

      테스트를 수행하는 동안에 테스터는 경험과 함께 실행할 새로운 좋은 테스트를 생성시키는 창의성을 습득함. 블랙 박스 테스팅 기법으로도 불림. 개발 프로세스의 특정 단계에서 특정 테스트 기법에 적용될 수 있는 테스트 접근 방식으로 고려됨. 핵심은 테스트 기법이나 테스트 항목, 검토가 아니라, 테스터의 인지적인 참여와 자신의 시간을 관리하는 책임성에 있음.

활동(Activity)명 6.3 인수 테스트
목적

시스템이 요구사항에 맞게 구축되었는가를 점검함. ‘알파/베타 테스트’가 수행되기도 함

  • 사용자/고객 관점에서 해당 기능이 올바로 구축되었는가를 점검함.
  • 사용자/고객에게 가치있는 시스템을 전달하기 위한 최종 점검을 수행함.
절차
  • 테스트 시스템 빌드 및 배포
  • 인수 테스트 수행

활동(Activity)명 6.4 테스트 결과 반영
목적

테스트 시행 중이나 테스트 종료 후 테스트 리포트에 포함된 결함의 원인이 되는 문제를 소스코드로부터 제거함

  • 시스템 기능 목표 달성
  • 소스코드 품질 목표 달성
  • 실행 시스템 품질 향상
절차
  • 테스트 준비, 시행, 완료 후 어느 시점이든 결함이 식별되면 수행함
  • 결함분석→ 담당자 식별 → 설계 검토 및 수정 → 코드 검토 및 수정 → 단위 테스트
역할자
  • 아키텍트, 설계자, 개발자
산출물
  • 테스트 결과 보고서에 포함(결함 조치 결과)
  • 수정된 설계 모델, 소스코드
