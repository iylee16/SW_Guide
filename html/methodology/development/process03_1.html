활동(Activity)명 4.1 서버 컴포넌트 설계
목적

업무 분석작업 결과로 얻은 도메인 객체 모델과 컴포넌트 명세서로 표현된 비즈니스 로직을 아키텍처 설계를 통해 얻은 기술구조에 올리는 작업을 수행함

  • 비즈니스 로직을 기술구조 속에 담는 작업을 수행함
  • 코딩 작업 전, 비즈니스 로직을 구조화 함
절차
  1. 인터페이스 설계
  2. 컴포넌트 내부 설계
  3. 컴포넌트 명세 상세화
역할자 시스템 분석가 또는 시스템 설계자
산출물

분석모델(분석 도메인 객체모델+컴포넌트명세모델(명세서))

POC 프로토타입 결과
가이드
  • 인터페이스 설계
    • 절차
      • 분석 단계 컴포넌트 리뷰
      • 변경 및 확장 요구 검토
      • 적용 기술 검토 및 데이터 타입 확인
      • 컴포넌트 인터페이스 작성
    • 가이드
      • 인터페이스 설계 결과는 가급적이면 모델 형태를 유지한다.
      • 감리, 리뷰 등의 목적으로 형식을 갖춘 인터페이스 명세서(또는 컴포넌트 명세서)가 필요할 경우, 템플릿을 기반으로 모델로부터 자동으로 문서를 생성하는 방법을 선택한다.
      • 매개변수는 여러 개의 원시(primitive) 타입을 사용하기 보다는 가급적 묶어서 하나 또는 두 개 정도의 객체를 사용한다. 이러한 방법은 매개변수 변경이나 확장에 대해 유연한 인터페이스를 유지할 수 있도록 해준다.
      • 오류 정보를 리턴하도록 설계하지 않고 예외(exception)와 같은 언어가 지원하는 오류처리 메커니즘을 활용한다.
      • 하나의 인터페이스를 지나치게 여러 가지 용도로 사용하지 않으며, 유시한 인터페이스를 많이 만들 지 않는다. 이 두 가지 요건은 상충 관계이므로 설계자는 누구나 쉽게 이해할 수 있는 명확한 원칙을 가지고 인터페이스 변형(variation) 이나 입도(granularity)를 조절 및 적용한다.
      • 모든 컴포넌트의 인터페이스를 설계하지 않는다. 유사한 경우는 분석 명세서로부터 바로 코드로 작성할 수 있으며, 필요 시 구현모델(reverse 결과 모델)을 참조용으로 사용한다.
  • 컴포넌트 내부 설계
    • 절차
      • 서비스 인터페이스 구현부분 구조 설계
      • 도메인 객체 부분 구조 설계
      • 지원 클래스를 위한 공간 마련(DTO객체, Key객체, Helper객체, Utility 객체 등)
    • 가이드
      • 분석결과와 아키텍처 설계 결과를 반영하여 업무컴포넌트를 설계함
  • 컴포넌트 명세 상세화
    • 절차
      • 설계 작업을 통해 추가 및 변경된 부분 추출
      • 분석 모델에 반영
    • 가이드
      • 컴포넌트 인터페이스 명세 작업 시 매개변수와 리턴 타입으로 가급적이면 객체를 사용한다.
      • 매개변수 객체나 리턴타입 객체들을 명시적으로 표현하기 위해 “인터페이스 정보 모델”을 작성한다. 이 모델은 인터페이스와 관련된 모든 정보를 표현한다.
      • 인터페이스는 계약(contract)의 한 종류로 보고 설계를 한다. 계약에는 다양한 조건들이 부과되며 이러한 조건들을 적절한 양식으로 표현을 한다. 단, 중요한 계약에 많은 부과 정보들이 추가되듯이 중요한 인터페이스에 대해서 인터페이스 정보모델을 작성한다.
활동(Activity)명 4.2 데이터 설계
목적

업무에 필요한 데이터를 분석하고 설계하여 시스템 구축과 운영에서 영속성을 갖는 업무 데이터를 식별하고 영속 메커니즘을 결정함

  • 업무가 어떤 데이터와 관련이 있는지 또는 데이터간의 관계는 무엇인지를 분석함
  • 업무에 필요한 데이터의 영속성과 무결성을 보장하기 위한 데이터 구조를 설계함
절차
  1. 코드설계
  2. 데이터전환 설계
  3. 논리데이터모델링
  4. 물리데이터모델링
역할자
  • 데이터베이스(업무) 설계자, 데이터베이스 관리자 (DBA)
산출물
  • 코드정의서: 업무 코드 정의서
  • 데이터모델: 논리모델, 물리모델모델
가이드
  1. 코드설계
    • 절차
      • 자주 사용하는 엔티티의 속성 식별
      • 유사 엔티티 속성 식별과 정련
      • 코드 엔티티타입(테이블) 설계
    • 가이드
      • 구분하기 위한 용도로 사용하는 엔티티 속성을 코드 후보로 식별한다.
      • 코드로 정의할 엔티티 데이터를 코드 엔티티(테이블)로 분리한다.
      • 코드 테이블이 수가 많고 각 테이블이 코드량이 적을 경우 유사한 구조의 코드 테이블을 통합니다.
      • 코드의 양이 많고 코드에 업무규칙이 수반되면 통합 코드에서 분리하여 별도의 엔티티로 정의한다.
      • 코드 테이블과 엔티티간의 관계를 정의한다.
      • 물리모델에서 코드 테이블과 엔티티간의 제약조건(FK)를 설정할지 정책을 결정한다. 제약조건을 설정할 경우 무결성은 안전하게 유지되는 반면 성능에 심각한 저하를 초래할 수 있다. 이는 코드 테이블의 량과 코드와 엔티티간의 관계의 수에 의해 정책을 결정할 수 있다.
      • 데이터베이스 전환이 필요한 경우 이전 AS-IS 업무 코드와 TO-BE 업무 코드간의 관계를 식별하고 데이터 전환설계에 정의한다.
  2. 데이터전환설계
    • 절차
      • 데이터전환 전략 수립 및 분석
      • 데이터전환 자료에 대한 정의
      • 데이터 전환(변환) 설계 및 개발
      • 데이터 정합성 검증 방안 준비
    • 가이드
      • 데이터전환은 이행방법 및 전환전략이 매우 중요하므로 데이터전환 계획서를 통한 전환전략을 명시한다.
      • 데이터전환의 범위는 AS-IS 데이터베이스가 될 수도 있으며 시스템 운영에 필요한 전자문서나 일반문서가 될 수도 있다.
      • 데이터전환의 범위와 전략에 따라 규모가 많이 변경될 수 있으므로 시스템 운영에 필요한 데이터 식별에 매우 유의한다.
      • 데이터전환 대상에 따라 전환전략을 세운다. 전환전략은 전환프로그램을 이용한 자동화, 전자문서 데이터베이스화 또는 일반문서 데이터베이스 등이 있으며 이에 따른 적절한 도구선정이 필요하다.
      • 데이터전환 일정은 시스템 개발에 많은 영향을 줄 수 있으므로 전체 개발 일정에 따라 단계적인 구축 전략을 취한다.
  3. 논리 데이터 모델링
    • 절차
      • 엔티티타입 정의
      • 관계정의, 식별자 및 속성 정의
      • 정규화, 비정규화
      • 도메인 및 용어사전 정의
    • 가이드
      • 유스케이스 명세서나 업무 기술서를 통해 영속성을 가져야하는 요소들을 식별한다.
      • 개념이 불분명한 것이나 광범위한 것은 제거한다.
      • 초기에 엔티티타입의 특성이나 속성값을 상세하게 정의하지 않는다.
      • 유스케이스 명세서의 시나리오를 통해 누락된 엔티티타입이 존재하는지 검증한다.
      • 유스케이슨 명세서나 업무 기술서를 통해 엔티티에서 발생할 수 있는 행위를 식별하여 관계를 설정한다.
      • 관계정의는 ERD나 UML 모델을 통해 엔티티 간의 관계를 도식화하여 검토한다.
      • 각 엔티티가 유일하게 구분할 수 있고 대표성을 가지는 식별자가 있는지 점검한다.
      • 복합 식별자를 가지는 경우 너무 많은 복합 식별자를 가지는지 점검한다.
      • 유스케이스 명세서나 업무 기술서를 통해 각 엔티티가 가져야 있는 속성을 식별한다. 또한 이를 통해 파생되는 속성을 식별하고 추가한다.
      • 파생 속성의 경우 계산 등에 의해 다른 속성 값에 영향 받는지 점검하고 속성간의 영향도를 최소화한다.
      • 모든 속성을 분류하여 공통의 특징을 식별하고 속성에 대한 도메인을 정의한다. 각 도메인별로 데이터 타입과 길이를 지정한다.
  4. 물리 데이터 모델링
    • 절차
      • 논리 모델 ERD를 물리 모델 ERD나 테이블 정의서로 전환
      • 테이블 정규화, 비정규화
      • 관계의 제약조건(FK) 설정 및 입력,수정,삭제 무결성 규칙 정의
      • 인덱스, 접근방법, 뷰, 파티션, 트랜잭션 설계
    • 가이드
      • 논리 데이터 모델의 엔티티 타입은 테이블, 주식별자는 PRIMARY KEY, 속성은 컬럼 그리고 엔티티 관계는 FOREIGN KEY로 변환한다.
      • 1:1, 1:M, N:M 의 관계를 물리모델에 맞게 변환한다. N:M의 관계는 물리모델에서 관계 테이블을 추가하여 변환할 수 있다.
      • 목적 데이터베이스 특성에 맞게 성능을 고려하여 비정규화가 필요한 부분을 식별한다.
        비정규화 시 데이터의 무결성을 해치지 않는 방안을 반드시 세워야 한다.비정규화는 테이블 병합이나 테이블 분할 그리고 테이블 추가에 의해 이루어진다.
      • 각 관계에서 데이터 무결성을 보장하기 위한 참조 무결성을 설계한다. 입력 참조 무결성, 수정 참조 무결성, 삭제 참조 무결성을 정의할 수 있다.
      • 과다한 참조 무결성은 데이터베이스 성능을 저하시키는 주요한 원인이 될 수 있다. 대용량의 입력,수정, 삭제가 필요한 테이블의 경우 참조 무결성 설계에 주의한다.
      • PRIMARY KEY, FOREIGN KEY에 의한 관계에 필요한 필요한 인덱스를 설계한다.
        인덱스 설계는 인덱스 컬럼, 순서의 식별 인덱스 명, 그리고 인덱스의 테이블스페이스 지정을 포함하여야 한다.
      • 업무분석을 통해 테이블의 데이터량을 추정하여 데이터베이스 용량을 설계한다. 추정된 용량을 근거로 물리 데이터베이스 파티션을 결정한다.
활동(Activity)명 4.3 UI 컴포넌트 설계
목적

UI기본원칙인 유용성, 사용성, 감성을 고려하여 사용자가 시스템을 효과적(effective)이고 효율적(efficient)으로 사용해 만족을 얻을 수 있도록 사용자 중심(user-centered)의 시스템 UI를 설계하는 활동임

  • 사용자가 손쉽게 시스템을 이용할 수 있도록 시스템의 전반적인 UI를 상세하게 설계한다.
  • 분석단계에서 획득하지 못한 상세한 요구사항을 시각적인 방법으로 고객에게 제공하기 때문에 요구사항을 추가 획득할 수 있고 정제할 수 있다.
  • 개발에 들어가기 전에 프로젝트 이해관계자와 시스템의 전체적인 UI 모습을 공유한다.
절차
  1. 화면구조설계(screen layout design)
  2. 화면설계(screen detailed design)
  3. 네비게이션설계(navigation design)
역할자

UI디자이너
산출물
  • UI 모델 또는 UI 설계문서(레이아웃 다이어그램(Layout diagram), 플로우 다이어그램(flow diagram), 네비게이션 다이어그램(navigation diagram) 포함)
가이드
  1. 화면구조 설계
    • 절차
      • 사용자의 화면구조에 대한 요구사항과 정황 분석
      • 화면구조 설계
      • 상호작용 설계(interaction design)
    • 가이드
      • 사용자의 정황과 메뉴의 깊이와 넓이, 정보의 양 등을 고려한다.
      • 화면 구조에 대한 최상위 레벨이므로 최상위 레벨에 위치할 요소가 무엇인지 정의한다.
      • 레이아웃에 배치된 메뉴 또는 링크가 다른 레이아웃에 어떤 영향을 주는지를 결정한다.
      • 레이아웃의 정확한 크기는 고려하지 않는다.(이는 비주얼 디자인을 할 때 그리드 시스템 방법을 적용하여 결정함)
  2. 화면 설계
    • 절차
      • 화면구성요소(기능, 컨텐츠, 입출력 항목 등) 식별
      • 화면 상세설계
      • 비주얼 컨셉 도출
      • 비주얼 디자인(화면, UI컴포넌트 등)
    • 가이드
      • 화면 상에서 수행될 핵심 비즈니스의 처리를 위한 요소는 사용자의 동선을 고려하여 배치한다.
      • 기능과 컨텐츠들 사이의 상대적 중요도를 고려하여 크기와 위치, 표현방식을 결정한다.
      • 정보의 양, 정보의 그룹, 정보의 정렬을 고려하여 배치한다.
      • 사용자의 언어나 업무에서 사용되는 실제 용어를 사용한다.
      • 개략적인 시나리오, 속성정보, 컨텐츠 설명, 주석 등을 스토리보드에 기술한다.
      • 결정되지 않은 기능과 컨텐츠에 대해서 임의로 설계를 할 경우 비주얼 디자인에 영향을 미칠 수 있으므로 결정되지 않은 것은 박스형태로 표현한다.
      • 화면설계는 개발 이전 화면의 최종 모습이므로 반복적으로 설계하고 검토한다.
      • 화면설계도구, 화면설계문서 템플릿 그리고 설계요소 엘리먼트를 사전에 협의한다.
  3. 네비게이션 설계
    • 절차
      • 유스케이스, 사이트 맵 분석
      • 화면, 화면 별 기능 및 이동접점 식별
      • 도식화된 네비게이션 다이어그램 작성
    • 가이드
      • 비즈니스를 처리하기 위한 화면, 화면 별 기능, 컨텐츠, 이동방식, 사용자 행위 등을 식별한다.
      • 화면에 들어갈 개략적인 기능과 컨텐츠는 의미 있는 또는 연관성이 있는 단위로 구분하여 화면을 분활한다.
      • 비즈니스 차원에서 연속 또는 상관 관계에 있거나 참조 관계에 있는 화면을 연결한다.
      • 일련의 절차에 의한 비즈니스 처리는 이동 화면의 개수, 단위 절차의 당위성, 페이지 별 정보의 양, 의미 있는 정보의 그룹핑 등을 고려하여 절차의 단계를 정의한다.
      • 화면 간 정보 흐름을 파악하기 위해 요소 간 연결고리를 표시하고 시나리오 또는 유스케이스를 기반으로 테스트를 수행한다.
      • 시작 포인트와 종료 포인트를 표현한다.
활동(Activity)명 4.4 외부인터페이스 설계
목적

아키텍처의 “외부 인터페이스 구조 설계”를 기반으로 목표 시스템과 외부 시스템의 인터페이스를 설계함. 전문 구성, 프로토콜 사용 설계 등을 포함

  • 명시적으로 설계함으로써 “Seperation of Concern”이라는 설계 원칙을 준수함
  • 외부로 나가는 모든 호출을 일목요연하게 설계 및 관리함
  • 외부로 나가는 메시지를 일관된 방식으로 설계 함
절차
  1. 아키텍처 팀의 도움을 받아 외부 인터페이스 구조 설계 이해 및 공유
  2. 전문구성방식 표준화
  3. 전문작성 및 인터페이스 로직 설계
역할자
  • 시스템 분석가 또는 시스템 설계자
산출물
  • 외부 인터페이스 설계(통합모델에 포함하며, 문서를 병행하여 사용할 수 있도록 함)
  • 특히 다른 부서나 회사 등과 교환을 하는 자료이므로 반드시 문서의 형태로도 관리해야 함
가이드
  • 명시적으로 설계함으로써 “Seperation of Concern”이라는 설계 원칙을 준수함
  • 외부로 나가는 모든 호출을 일목요연하게 설계 및 관리함
  • 외부로 나가는 메시지를 일관된 방식으로 설계 함
  • 전문 내부 속성은 확장성을 염두에 두고 결정하여야 함. 그렇지 않을 경우 전문 변경에 따른 작업 부하가 증가함
  • 인터페이스 하는 두 시스템 간의 거리가 멀 수로 Xml 등과 같은 유연한 형태의 전문 표현 방식을 사용하여야 함
  • 전문과 프로토콜 간에 의존성을 최대한 줄이는 방식으로 설계함
