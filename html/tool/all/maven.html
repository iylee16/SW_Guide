도구 소개

도구명 Maven
(http://maven.apache.org/)		 라이센스 Apache License, Version 2.0
소개 자바 기반 프로젝트를 빌드하고, 구성요소 및 라이브러리 의존성을 관리하는 도구이다.
특징
  • 프로젝트에 필요한 라이브러리를 POM 파일 만으로 쉽게 구성이 가능하다.
  • Convention over Configuration 개념으로 일반적인 프로젝트 폴더 구조를 사용한다.
  • Jenkins 같은 CI 도구와 연계하여 빌드 과정 자동화가 가능하다.
주요 기능
  • 빌드 로직의 재사용 및 빌드 관리
  • 의존성 관리
  • 원격 리파지토리 제공
  • 빌드, 패키지, 로컬 설치, 원격 배포
실행 환경 Windows, Linux, MacOS 사전 설치 도구 JDK 1.7 이상K
카테 고리 빌드 최신 버전 v3.5.4 (2018.06.21)
관련 도구
  • Ant (http://ant.apache.org/)
  • Gradle (https://gradle.org/)
  • Jenkins (https://jenkins.io/)

주요 기능

JAVA 기반 프로젝트 빌드 관련 자동화 도구

원격 저장소를 통한 라이브러리 및 클래스 패스 자동 구성

라이브러리 의존성 관리

설치 환경

windows, Linux, MacOS 별로 별도의 설치 파일 다운 및 설치를 진행한다.

사전 JDK 1.7이상의 도구 설치 필요(https://www.oracle.com/technetwork/java/javase/downloads/index.html)

설치 단계

1. Maven 다운

다운로드 사이트 : http://maven.apache.org/download.cgi

2. 시스템 등록

다운받은 maven 파일 압축 해제 후 시스템 등록을 한다.

  • 제어판 → 시스템(단축키 : windows key + break) → 고급 → 환경변수

신규 시스템 변수 등록

  • 새로만들기 → 변수명 (MAVEN_HOME), 변수값 (압축 해제 경로)을 입력한다.

사용자 변수 편집

  • 새로만들기 → 변수명(PATH), 변수값(%MAVEN_HOME%\bin)

※ PATH가 사전에 있을 경우 기존 항목 뒤 ;%MAVEN_HOME%\bin을 입력한다.

3. 설치 확인

windows 명령창 (windows key + R → cmd) 열기 → mvn -version을 실행한다.

maven과 java platform, 운영체제 명 및 버전을 확인한다.

Template Java application 프로젝트 생성

각 프로젝트의 성격에 맞도록 Template 프로젝트를 제공 : archetype

  • 프로젝트의 생성
    • 시작 → 실행 → “cmd”입력 → 사용자가 원하는 작업공간 디렉토리로 이동한다.
    • Prompt에서 “mvn archetype:create -DgroupId=[groupId]-DartifactId=[artifactId]”을 입력한다.

      ※ 본 매뉴얼은 groupId를 SMU로 지정, artifactId를 gbuddhajavapp로 정의한다.

    • 프로젝트 생성 후 디렉토리 구조 확인

Template Web application 프로젝트 생성

웹 어플리케이션 개발을 위한 template Archetype을 제공한다.

  •  Prompt에서 “mvn archetype:create -DgroupId=[groupId] -DartifactId=[artifactId] -DarchetypeArtifactId=maven-archetype-webapp ”을 입력한다.
    • Archetype의 지정을 위해 “DarchetypeArtifactId=maven-archetype-webapp”를 추가한다.
    • 프로젝트 생성 후 디렉토리 구조

POM 설정 파일

POM 설정 파일 : Archetype을 이용하여 Maven 기반 프로젝트 생성 시 필요하다.

  •  Project
    • pom.xml파일의 최상위 엘리먼트
    • modelVersion
      • POM model의 버전
    • groupId
      • 프로젝트를 생성하는 조직의 고유 아이디, 일반적으로 Full도메인
    • artifactId
      • 해당 프로젝트에 의하여 생성되는 artifact의 고유 ID
      • Maven을 이용하여 pom.xml을 빌드할 경우
        [artifactid]-[version].[packaging]의 규칙으로 artifact가 생성. 예시의 경우 빌드 시 gbuddhajavapp-1.0-SNAPSHOT.jar 파일이 생성된다.
    • Packaging
      • 해당 프로젝트를 어떤 형태로 packaging할 것인지 결정한다.(jar, war, ear 등)
    • Version
      • 프로젝트의 현재 버전, 개발 중 일시, SNAPSHOT을 접미사로 사용한다.
    • Name
      • 프로젝트 이름
    • url
      • 프로젝트 사이트가 있는 경우 사이트 URL 등록이 가능하다.
    • Dependency
      • 현재 생성한 프로젝트와 의존관계가 있는 프로젝트의 패키지 정보

Compile

프로젝트를 build하고 관리 - 컴파일 build phase : compile

  • Maven에서 여러 종류의 Build Phase들을 지원하며 관리한다.
  • 콘솔 창에서 해당 프로젝트의 위치로 이동 → “mvn compile”입력
    • Compile 명령은 프로젝트의 디렉토리에서 “src\main\java”에 위치한 코드만을 컴파일한다.
    • Compile 후에는 프로젝트 디렉토리 아래 target 디렉토리가 생성된다.

test-compile

테스트코드의 컴파일 - build phase : test-compile

  • 콘솔 창에서 해당 프로젝트의 위치로 이동 → “mvn test-compile”을 입력한다.
    • test-compile명령은 프로젝트의 디렉토리에서 “src\test\java”에 위치한 코드만을 컴파일한다.
    • test-compile후에는 프로젝트 디렉토리 아래 target\test-classes디렉토리가 생성된다.

test

구현코드를 테스트 - build phase : test

  • 콘솔 창에서 해당 프로젝트로 이동 → “mvn test”입력
    • Test명령은 프로젝트의 디렉토리에서 “target\test-classes”의 junit의 단위 테스트 클래스를 이용하여 “target\classes”에 위치한 구현 클래스를 테스트한다.
    • Test한 결과 정보는 프로젝트 디렉토리 아래 target\surefire-reports디렉토리에 생성된다.

package

구현 코드를 압축(jar, war) - build phase : package

  • 콘솔 창에서 해당 프로젝트로 이동 →“mvn package”입력
    • Package명령을 실행하면 compile, test-compile, test, package순으로 명령이 실행 → jar, war 파일이 target 디렉토리에 생성된다.
    • Pom 파일의 artifactId, version, packaging 정보가 package파일의 파일명으로 지정

      ※ 생성된 jar, war의 파일명 형식은 [artifactId]-[version].[packaging]

Install

Local Repository에 등록 - build phase : install

  • 콘솔 창에서 해당 프로젝트로 이동 →“mvn install”을 입력한다.
    • compile, test-compile → test → package → install → Local Repository에 프로젝트 등록
    • 윈도우 기본 지정 위치 : “C:\Documents and Settings\[사용자이름]\.m2\repository\[groupID]”

Deploy

Package파일을 Remote Repository에 등록 - build phase : deploy

  • compile → test-compile → test → package → install → deploy → Remote Repository에 등록
    • emote Repository의 설정 : pom 파일 추가하여 설정 가능 : Distribution : 설정 섹션 생성 필요
      • id : 저장소의 아이디 설정
      • name : 저장소의 이름을 결정
      • url : 저장소의 url을 지정

Dependency Library 관리

애플리케이션 개발 시 기존의 외부 라이브러리 관리 방법의 문제점

  • 라이브러리의 추가 및 버전 변경이 불편하다.
  • 버전관리시스템(CVS, SVN)을 이용하여 관리할 때 공유할 파일 크기가 커진다.
  • 현재 사용하고 있는 라이브러리의 버전을 파악하기 힘들다.
  • 컴파일, 배포, 테스트할 때만 사용하는 라이브러리를 분리하기 힘들다.
  • WTP(Web Tool Platform)의 경우 자동 클래스 패스 기능이 너무 느리다.

기존의 외부 라이브러리의 수정 과정

  • 라이브러리간의 의존도가 많아지면서 라이브러리의 수정이 더욱 힘들어진다.

Maven의 의존성 관리 : POM 설정 파일

  • POM 설정 파일의 <dependencies>섹션을 변경하여 관리
  • pom.xml파일을 아래와 같이 설정 → “mvn compile” 명령 실행한다.
    • junit 3.8.1 버전의 jar파일이 자동으로 Local Repository에 다운받는다.
  • Jar파일이 관리되는 원칙 : Maven Local Repository의 변경 없이 디폴트 디렉토리를 사용하고 있다면 USER_HOME]\.m2\repository 디렉토리에 모든 Dependency 라이브러리를 다운받는다.
    • 윈도우의경우USER_HOME은“C:\Documents and Settings\[사용자아이디]”
  • 아래와 같이 pom.xml 파일을 설정 → 다음 위치에 라이브러리의 jar 파일이 다운된다.
    • USER_HOME\.m2\repository\junit\junit\3.8.1\junit-3.8.1.jar
  • 라이브러리를 관리하는 디렉토리의 규칙 :
    • USER_HOME\.m2\repository\[groupId]\[artifactId]\[artifactId-version.jar]
  • 라이브러리의 버전 변경은 pom.xml파일에서 <version>부분을 수정하여 변경한다.

라이브러리의 성격에 따라 각 라이브러리의 Scope 지정이 가능하다.

  • 기본적으로 프로젝트의 생성 시 지원되는 junit은 단순히 테스트를 목적으로만 사용되는 라이브러리
    • pom.xml 파일에는 junit의 dependency설정 내부의 scope설정 : test
  • Scope설정 및 기능
    • compile : 기본값은 scope, 컴파일 시 사용, 배포 시에도 같이 배포되어야 하는 라이브러리
    • provided : JDK가 컨테이너에 의하여 제공되는 라이브러리
      • servlet.jar의 경우 Servlet 컨테이너에 의하여 제공되기 때문에 provided scope를 사용한다.
    • runtime : 컴파일 시에는 사용되지 않지만 애플리케이션을 실행할 때 사용되는 라이브러리
      • test : 테스트를 위해서만 사용하는 라이브러리
      • system : provided와 비슷하지만, 직접 jar파일을 제공해야 한다.
      • 단, repository에서 관리되지 않을수도 있다.

의존 관계 중 특정 라이브러리 배제 기능<exclusions>

  • 사용자가 사용할 라이브러리X가 라이브러리A, 라이브러리B에 의존성이 있을 경우, pom.xml 파일에는 X만 명시해도 A와 B는 Maven이 가져온다.
  • X의 A에 대한 의존성이 버전 문제 등으로 제대로 동작하지 않을 수 있는데, 이 경우 <exclusions> 태그를 사용해서 A를 배제하고 사용자가 직접 A를 가져와서 의존성 문제 해결이 가능하다.

부모 프로젝트의 버전 범위 지정

  • 프로젝트X가 다른 프로젝트인 프로젝트A에 의존성이 있을 경우, <parent>를 통해 A의 설정 사항을 X가 상속받을 수 있다.
  • 이 때, A의 특정 버전 범위를 명시해서, 해당 조건을 만족하는 경우에 대해서만 상속 받을 수 있도록 설정 가능하다.

인코딩 지정

Eclipse Plug-in 활용

Eclipse에서의 프로젝트 생성

  • 프로젝트 디렉토리로 이동 → "mvn eclipse:eclipse" 명령어 실행 → Eclipse기반 프로젝트로 변경된다.
    • Eclipse프로젝트에 필요한 .project, .classpath 파일들이 자동적으로 생성된다.

Java application의 Eclipse프로젝트를 생성

  • .classpath 파일에 pom.xml파일에 정의된 의존 관계에 있는 라이브러리가 자동으로 추가된다.
    • 라이브러리를 배포할 때 자바 소스까지 분리해서 배포하는 것이 가능하도록 지원한다.
    • 기존의 외부 라이브러리에 대한 소스 코드가 존재하지 않아 소스코드를 분석하는데 한계가 있었지만, Maven을 이용하여
      • 소스 코드를 배포하는 모든 라이브러리의 소스 코드는 자동적으로 Local Repository에 다운받아지기 때문에 Eclipse에서 외부 라이브러리 소스코드 접근이 용이하다.

Build Phase

Build Phase를 이용하여 각 Build Phase의 목적에 맞는 Goal을 설정

build phase의 lifecycle

  • Clean lifecycle
    • pre-clean : 프로젝트의 제거 전에 필요한 과정을 수행
    • Clean : 이전 빌드로 생성된 모든 파일을 제거
    • post-clean : 프로젝트 제거의 마무리 과정을 수행
    • Package lifecycle
    • Compile : 프로젝트의 소스코드를 컴파일
    • est-compile : test 소스코드를 컴파일
    • Test : test framework를 이용하여 test를 수행, Package나 deploy시에는 필요 없음
    • Package : 프로젝트를 jar, war, ear등 package파일로 생성
    • Install : local repository에 해당 package를 설치
    • Deploy : 원격의 repository에 패키지를 다른 개발자와 공유할 수 있도록 설치
  • Site lifecycle
    • pre-site : 프로젝트의 사이트를 생성하기 전의 과정을 수행
    • Site : 프로젝트 사이트 문서를 생성
    • post-site : 프로젝트의 사이트 생성의 마무리 과정을 수행하고 사이트 배치를 준비
    • site-deploy : 웹 브라우저에서 프로젝트의 사이트를 확인할 수 있도록 사이트를 배치
